1. Используем максимально закрытые accessor-ы:
    1. public только когда есть внешний пользователь
    2. protected когда нет внешнего пользователя, но есть наследование
    3. private по умолчанию
  2. Каждому новому классу, свойству и методу даём краткое описание
    1. Когда описание не требуется:
      1. когда в качестве свойства используется объект класса с уже существующим описанием
    2. Постарайтесь выразить предназначение класса/метода в одном предложении без использования "и" и "или". Если не получается - значит что-то с вашим классом не так.
  3. В phpdoc не стесняемся вместо array писать из чего состоит массив, например так: int[]
  4. typehint обязателен как для аргументов, так и для возвращаемых значений
  5. Мы соблюдаем контракты. Если метод обязался возвращать int, то он всегда должен возвращать int.
  6. Все должны быть объявлены/созданы в предназначенных для этого местах, либо как аргументы конструктора.
    1. Исключение можно сделать только для старых классов, которые хрен знает где создаются. Там в конструкторе используем app(Class::class).
  7. Мы не используем магические числа! Если надо указать какое-то число в коде, то оно должно быть либо в константе, либо подписано.
    1. Для дней, часов и минут есть константы тут: \App\Helpers\DateTimeHelper::DAY
  8. Если в коде необходимо использовать название таблицы, то берём его из TABLE_NAME модели.
  9. Старайтесь не изобретать велосипед. У Laravel есть прекрасная документация.
    1. Зачастую вижу недостаток знаний доступных возможностей в коллекциях, смотреть можно тут
  10. Нам не нужен неиспользуемый код. Пожалуйста, удаляйте за собой.
  11. Для запросов используем Repositories (app/Repositories)
  12. При создании реквеста обращаем внимание на галочку CI (пункт phpstan должен отработать)"
  13. Не забываем передвигать тикет в асане по этапам
