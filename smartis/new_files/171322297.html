Функционал фильтрации Smartis

Задачей фильтра является исключение или включение определенных данных по их прямым свойствам. Для разных показателей могут быть доступны разные фильтры (так как у разных данных могут быть разные свойства). Фактически фильтр применяется к выбранным показателям (к которым его возможно применить в связи с особенностью их данных).

Фильтры доступны в конструкторе отчетов, в некоторых дашбордах некоторых клиентов и в разделе настройки показателей. Работают аналогично в обоих разделах.

О применении фильтров при настройке показателей читать тут: 4.3. Раздел «Показатели»: создание и настройка показателей, работа с фильтрами

О применении фильтров в дашбордах читать тут: 

Чтобы добавить фильтр к отчету, нужно нажать "+" (как на скрине)


В выпадающем списке можно ввести название фильтра либо выбрать, прокрутив вниз.

После выбора фильтра появится окно выбора условия фильтрации: чаще всего это равно, не равно, содержит, не содержит, соответствует регулярному выражению, не соответствует регулярному выражению, пустое значение, не пустое значение.


Фильтры могут применяться на 3 уровнях:

  • в показателе (вшитые в показатель фильтры)
  • в отчете (в конструкторе отчетов)
  • в дашборде

У этих уровней есть своя иерархия:

По одному и тому же фильтру с разными условиями фильтрация в отчете (в дашборде) будет первостепеннее фильтрации, "вшитой" в показатель, поэтому условие, которое "вшито" в показатель игнорируется.

Пример: показатель "Бронь бесплатная" настроен с помощью вшитых фильтров так, чтобы отображать в отчете только бесплатные бронирования

Строим отчет с этим показателем и добавляем к отчету фильтр с условием по платным броням. Вид брони = платный.

В отчете отображаются данные по платным броням, хотя на уровне показателя такие брони мы исключаем.

Некоторые фильтры в Smartis можно добавить к отчету или показателю несколько раз. К таким фильтрам относятся:

  • Когорта персон
  • Логика подсчета значений показателя
  • и другие. Если вам нужно для фильтра по доп.полям CRM вывести значения, например Площадь больше Х, но меньше Y, то для решения такой задачи также нужно использовать фильтр дважды. Если ваш фильтр нельзя применить дважды, то необходимо поставить задачу в консультацию на добавление возможности применять нужный фильтр больше 1 раза. 

Виды фильтров в Smartis

  1. Общие фильтры - системные фильтры Smartis, которые применяются к данным разных типов. Пример таких фильтров:

    Номер телефона клиента - фильтрует данные по номеру телефона, указанному в формате 79998887766
    Email клиента - фильтрует по email клиента
    Yandex Client ID - фильтрует по yandex client id Яндекс Метрики
    Google Client id - фильтрует по id Google
    Ссылающаяся страница (referrer) - по адресу ссылающейся страницы
    Страница входа (посадочная страница) - по адресу посадочной страницы
    Исходное обращение (parent_request_id) - по parent_request_id обращения. 

    • условие "равно" конкретная цифра - означат что ищем дочерние звонки у которых родитель обращение с этим parent_request_id

    • не равно - ищем все звонки у которых родитель не этот parent_request_id (дочерние и другие родительские)

    • пустое значение - выберутся все родительские звонки без технических дублей

    • не пустое значение - выберутся все звонки, которые дочерние + тех дубли

    • в списке через запятую - означат что ищем дочерние звонки у которых родительские обращения с этими parent_request_id
    • не в списке через запятую - ищем все звонки у которых родитель НИ ОДИН ИЗ этих parent_request_id (дочерние и другие родительские)

    Час события - фильтрует по времени, когда произошло событие (указанное в выбранном показателе)
    Когорта телефонов - строит когортный отчет от данных по обращениям ( описано тут)

  2. Фильтры для данных по посещениям сайта - фильтруют данные показателя Посетители Smartis и его производных по признакам, которые доступны в данных о сессиях на сайте клиента. Список таких фильтров:
     

    • Smartis.Analytics: CallTouch ID сессии - фильтрует по CallTouch ID сессии которую присваивает CallTouch (если подключен у клиента)

    • Smartis.Analytics: CoMagic ID сессии - фильтрует по Comagic ID сессии которую присваивает Comagic (если подключен у клиента)

    • Smartis.Analytics: GetResponse CONTACT_ID (gr_sid) - фильтрует по GetResponse CONTACT_ID сессии которую присваивает GetResponse (если подключен у клиента)

    • Smartis.Analytics: IP адрес пользователя - фильтрует по IP адресу

    • Smartis.Analytics: p_id (идентификатор браузера) - по идентификатору браузера пользователя

    • Smartis.Analytics: Smartis fingerprint ID [beta] -  для пользователей смартис сейчас не используется.

    • Smartis.Analytics: smid - для ROPO через смс, позволяет наблюдать, как люди переходят по ссылкам из смс. Для поиска сессий, у которых заполнены параметры smid

    • Smartis.Analytics: Браузер - по названию браузера пользователя

    • Smartis.Analytics: Наименование робота - по названию робота, который совершил сессию

    • Smartis.Analytics: номер посещения в рамках сайта - можно указать, сессии с каким по счету посещением в рамках выбранного сайта вывести в отчет.

    • Smartis.Analytics: номер посещения по всем сайтам аккаунта - можно указать, сессии с каким по счету посещением в рамках всех сайтов клиента, на которых установлен код Smartis, вывести в отчет.

    • Smartis.Analytics: Операционная система устройства пользователя - выбрать сессии, совершенные пользователями определенных операционных систем.

    • Smartis.Analytics: посетитель - робот? (1/0) - позволяет исключить или выбрать роботов, которые совершали сессии на сайты клиента.

    • Smartis.Analytics: Причина создания сессии - выбрать сессии с определенными причинами создания.

    • Smartis.Analytics: Продолжительность сессии, с - выбрать сессии с определенной продолжительностью.

    • Smartis.Analytics: Сессия - внутренний переход? (1/0) - исключить или включить сессии, которые являются внутренними переходами.
      Глубина просмотра - выбрать сессии с определенной глубиной просмотра.
      Устройство пользователя - выбрать сессии по типу устройства (ПК, планшет, мобильный телефон)



  3. Фильтры для данных конкретных сервисов - фильтруют только данные сервиса, который указан в названии фильтра. Для каждого сервиса набор фильтров отличается и зависит от данных, которые этот сервис передает. Отличить такие фильтры можно по их названию, например:

    CoMagic: Статус звонка
    Яндекс Директ: Тип площадки


  4. Фильтры по источникам трафика - фильтруют данные, которые содержат признаки источника трафика. Список таких фильтров:

    • utm_campaign - фильтрация по данному параметру utm

    • utm_content - фильтрация по данному параметру utm

    • utm_medium - фильтрация по данному параметру utm

    • utm_source - фильтрация по данному параметру utm

    • utm_term - фильтрация по данному параметру utm
      Источники в Smartis: Канал - фильтрация по выбранным каналам
      Источники в Smartis: Ключевое слово - фильтрация по выбранным ключевым словам (для контекстной рекламы)
      Источники в Smartis: Наименование кампании - фильтрация по названию кампании (как она называется ы Smartis)
      Источники в Smartis: Наименование объявления - фильтрация по названию объявления (для источников, по которым есть данные по объявлениям)
      Источники в Smartis: Площадка - фильтрация по выбранным площадкам

      Источники в Smartis: Группа объявлений - Позволяет отфильтровать данные по наименованию группы объявлений. 
      Применима только для площадок Яндекс/ВК/Майтаргет или Facebook.

      Фильтр применим к данным таких типов:

      • данные из рекламных кабинетов (клики, показы, расход, лиды)
      • посетители Смартис (сессии)
      • все обращения (звонки, заявки, чаты, анкеты, квизы и прочие)
      • CRM данные
      Работает по условиям равно, не равно, включает, не включает.

    • Источники в Smartis: Аккаунт рекламной системы - Позволяет фильтровать данные по наименованию рекламного кабинета, из которого приходят данные (это название аккаунта, отображаем именно названия кабинетов, а не почту через которую подключена интеграция). 

      Применима только для площадок Яндекс/ВК/Майтаргет или Facebook.

      Если у вас подключена интеграция в смартис через "общий" аккаунт, у которого есть доступы к нескольким аккаунтам внутри рекламного кабинета, то в фильтре будут названия из рекламного кабинета по всем логинам. 

      Группировка применима к данным таких типов:

      • данные из рекламных кабинетов (клики, показы, расход, лиды)
      • посетители Смартис (сессии)
      • все обращения (звонки, заявки, чаты, анкеты, квизы и прочие)
      • CRM данные

      Если к аккаунту нет доступа - по нему можно будет фильтровать, в отчете отобразятся архивные данные по нему, которые мы получили ДО момента потери доступа.

      Если аккаунт отключен - по нему можно будет фильтровать, в отчете отобразятся архивные данные по нему, которые мы получили ДО момента отключения логина.

      Игнорируем только УДАЛЕННЫЕ аккаунты - если аккаунт удален, то данные по нему нельзя будет отфильтровать. 
      Условия фильтрации: включает, не включает, равно, не равно. Кабинеты можно выбирать из выпадающего списка. 


  5. Фильтры для платных функционалов (Верификация, CDP) - фильтруют данные, которые обрабатывает такой функционал. Список фильтров:

    • Верификация: Выходит клиент на связь после обращения?  - фильтрует только верифицированные обращения по признаку выхода клиента на связь после обращения

    • Верификация: Детали - фильтрует только верифицированные обращения, можно указать текст или часть текста из поля "детали" у верифицированных обращений

    • Верификация: Есть причины отказа? - фильтрует только верифицированные обращения по наличию причин отказа, установленных алгоритмом верификации для клиента

    • Верификация: Есть продвижение по воронке? - фильтрует только верифицированные обращения, которые продвинулись-не продвинулись по воронке (что считать продвижением - прописывается в алгоритме верификации клиента)

    • Верификация: Есть целевые теги? - фильтрует только верифицированные обращения по наличию целевых тегов (их перечень прописывается в алгоритме верификации клиента)

    • Верификация: Итог - фильтрует обращения по различным итогам верификации (целевое, не целевое, условно целевое и тд)

    • Верификация: Клиент брошен? - фильтрует только верифицированные обращения, позволяет найти брошенных клиентов (что такое брошенный клиент - определяет алгоритм верифиации клиента)

    • Верификация: Клиент зарегистрирован в CRM? - фильтрует только верифицированные обращения по призанку есть ли клиент в CRM или нет

    • Верификация: Объект в СRM (интереса подтверждающего целевое) - фильтрует только верифицированные обращения по признаку объекта в CRM (какой объект подтверждает, что обращение целевое - определяет алгоритм верификации клиента)

    • Верификация: Оспоренность - фильтрует только верифицированные обращения, можно посмотреть итог верификации каких обращений был оспорен (оспаривать можно каждое верифицированное обращение индивидуально вручную)

    • Верификация: Первичность обращения - фильтрует только верифицированные обращения по признаку первичности

    • Верификация: Период причин отказа - фильтрует только верифицированные обращения, можно выбрать определенные период в течение которого получили отказ (причины отказа указаны в алгоритме верификации клиента)

    • Верификация: Подтвержденность вручную - фильтрует только верифицированные обращения, можно выбрать обращения которые подтверждены/не подтверждены вручную

    • Верификация: Продвижение до брони - фильтрует только верифицированные обращения, можно выбрать обращения, котоыре продвинулись/не продвинулись до бронирования. Что считать продвижением - указано в алгоритме верификации клиента.

    • Верификация: Тип объекта в СRM (интереса подтверждающего целевое) - фильтрует только верифицированные обращения по признаку типа объекта в CRM (какой объект подтверждает, что обращение целевое - определяет алгоритм верификации клиента)
      Фрод: процент вероятности       - доступно только для клиентов с решением Антифрод и позволяет выделить обращения или CRM элементы с указанным процентом вероятности фрода.
      Сегмент - доступно только для клиентов с решением Конструктор сегментов и позволяет выделить персон, которые попадают/не попадают в указанный сегмент
      Сегмент на дату попадания в сегмент - доступно только для клиентов с решением Конструктор сегментов и позволяет выделить персон, которые попадают/не попадают в указанный сегмент, персоны выделяются с учетом даты попадания в сегмент (какой период выбран в отчете, за такой период ищем персон, которые попали в этот сегмент)
      Сегменты: Инструменты Матчинга (ROPO и другие)
      Сегменты: Интерес к проектам - доступно только для клиентов с решением Конструктор сегментов, позволяет выделить персон, которе между несколькими объектами (ЖК, дилерскими центрами) отдают предпочтение указанному (ЖК или дилерскому центру)



    1. Фильтры по календарю - фильтруют данные по заданным условиям выбора дат. По умолчанию (без применения фильтров по календарю) данные строятся на дату их создания. Список таких фильтров:



      • Календарь: Дата отображения в отчете - меняет период, в котором будут отображаться данные, опираясь не на дату создания элемента, а на другую выбранную из списка дату (например, дату закрытия или любую другую). 
        Какие задачи может решать: 

      • - клиент учитывает сделки только когда они закрыты (то есть у них заполнена дата закрытия)
        - клиент учитывает отказы только когда срм элемент переходит в статус отказа (заполнена дата отказа) и т.д.

        Пример к задаче "Учитывать сделки как успешные в том месяце, когда они были закрыты, а не созданы": Сделка создана в CRM в августе. У этой сделки в CRM указана также дата закрытия = октябрь (ее закрыли в октябре). Добавляем фильтр "Дата отображения в отчете" Равно "Дата закрытия". В отчет мы выведем сделку не в августе, а в октябре.
        То есть фильтр меняет отображение в отчете по периоду: если есть договор, у которого дата закрытия = октябрь, то в отчете такой договор будет отображаться в октябре. С таким фильтром в выбранном периоде будут показаны те договоры, у которых дата закрытия находится в выбранном периоде. Если какой-то конкретный договор не попал в отчет - проверьте, какая у него дата закрытия - скорее всего, она в другом периоде.

        Как это выглядит в Smartis:
        Показатель "Кол-во сделок" + фильтр Календарь: Дата отображения в отчете равно Дата подписания договора. Видим, что договор отображается 31.08.2023

        В детализации видим - Дата создания - 24.08, Дата подписания - 31.08. Все отображается согласно установленному фильтру.




      • Календарь: Дата фильтрации периода         - позволяет выводить в отчет данные, у которых установленная дата находится в выбранном периоде. При этом дата (по дням) а которой отображается в отчете срм элемент, не изменится.
        Фильтр позволяет выделить элементы, которые перешли на нужный этап в выбранном периоде, но при это они могли быть созданы в любом прошедшем периоде. (аналогично фильтру Амо дата перехода на этап). то есть данный фильтр позволяет игнорировать дату создания элемента, даже если она была не в выбранном периоде.

        Пример использования:
        - вывести в отчет все сделки, которые были закрыты в выбранном периоде
        - вывести в отчет все договоры купли-продажи, которые были оплачены в выбранном периоде.

        Как это выглядит в Smartis. На примере "посмотреть сделки АМО, у которых дата визита в сентябре"
        Пример разберем на 1 сделке для простоты понимания: мы выбрали 1 id сделки и построили по нему отчет без фильтров, период - сентябрь.

        Добавим фильтр по дате визита 

        Сделка теперь не попадает в отчет, тк у нее дата визита не в сентябре: дата визита 20 октября 2023 года.


      • Чем фильтр Календарь: Дата отображения в отчете отличается от Календарь: Дата фильтрации периода: фильтр по дате отображения изменяет дату, в которой отображается в отчете срм элемент. Дата фильтрации не меняет дату, в которой отображается срм элемент, но если искомая дата находится не в выбранном периоде, то в отчет этот элемент не попадет.

Например, колл-трекинг поставил, что этот звонок был 5м. Это и есть тот самый порядковый номер обращения. Соответственно, колл-трекинг четко поставил свой номер заявке/звонку. Тот номер, который проставил сервис обращений, мы и укажем в данных (и по нему же будем фильтровать). То есть, мы здесь ничего не придумываем, никакой логики не добавляем. Далее уже возможности фильтрации будут определяться тем, передает ли нам внешний сервис обращений этот параметр или нет. если не передает - необходимо уточнять почему у сервиса (не у нас).

    Обратите внимание, что при использовании данного фильтра НЕЛЬЗЯ вывести в отчет обращения с указанным порядковым номером относительно CJM в Смартис (например, отчет по первым обращениям в истории персоны). Это связано с тем, что описано выше - свойство передается из сервиса обращений. То есть, если в колл-трекинге это первое обращение от клиента, то номер будет = 1. При этом, если этот же клиент обратится к застройщику через чаты/заявки в другом сервисе, и в нем его обращение тоже будет первым, то и этому обращению будет присвоен номер 1. В смартис мы воспроизведем номера, которые передадут сервисы и сами НЕ будем выявлять, какое из обращений первое в истории персоны среди всех ее обращений из всех сервисов.

Способы фильтрации

У разных фильтров разные условия фильтрации ("по какому условию фильтровать") и разные способы выбора параметров фильтрации ("что указано в качестве критерия фильтрации")
Условие фильтрации выбирается после того, как выбран фильтр

Далее выбирается параметр фильтрации

Набор условий и параметров фильтрации зависит от выбранного фильтра

Для большинства фильтров доступны эти условия фильтрации (у некоторых фильтров их может быть меньше):
Равно - полное соответствие
Не равно - полное не соответствие
Включает - можно выбрать несколько параметров, по которым будет фильтрация. Это условие доступно для фильтров, у которых параметр фильтрации невозможно ввести вручную (выбирается из списка)
Не включает - можно выбрать несколько параметров, которые будет исключать фильтрация. Это условие доступно для фильтров, у которых параметр фильтрации невозможно ввести вручную (выбирается из списка)
Содержит - в искомых данных есть часть, которую указали в фильтре. Это условие доступно для фильтров, у которых параметр фильтрации вводится вручную

Не содержит - в искомых данных нет части, которую указали в фильтре. Это условие доступно для фильтров, у которых параметр фильтрации вводится вручную
Соответствует регулярному выражению - соответствует условиям, прописанным регулярным выражением. Это условие доступно для фильтров, у которых параметр фильтрации вводится вручную
Не соответствует регулярному выражению - не соответствует условиям, прописанным регулярным выражением. Это условие доступно для фильтров, у которых параметр фильтрации вводится вручную
Пустое значение - выбранный параметр не заполнен в данных. При использовании такого условия параметр фильтрации не заполняется
Не Пустое значение - выбранный параметр заполнен в данных. При использовании такого условия параметр фильтрации не заполняется
В списке через запятую - включает значения, указанные в списке через запятую. Это условие доступно для фильтров, у которых параметр фильтрации вводится вручную
Не в списке через запятую - не включает значения, указанные в списке через запятую. Это условие доступно для фильтров, у которых параметр фильтрации вводится вручную

*Если для какого-то фильтра нужно добавить условие фильтрации, обратитесь в тех.поддержку Smartis.

Для фильтров с числовыми значениями доступны такие условия фильтрации:
Больше - вывести в отчет данные, которые больше заданного параметра фильтрации
Меньше- вывести в отчет данные, которые меньше заданного параметра фильтрации
Равно - вывести в отчет данные, которые равны заданному параметру фильтрации
Не равно - вывести в отчет данные, которые не равны заданному параметру фильтрации

Пример применения циклов: нужно построить цикл от Обращения до сделки. Цикл измеряется в днях и показывает кол-во дней, которое занимает прохождение клиента от даты первого указанного события до даты второго указанного события.

Например, чтобы расчитать сколько дней прошло между датой создания сущности в CRM до даты встречи (факт), мы используем показатель "Встречи" и к нему добавляем 2 фильтра:
Логика подсчета показателя - Цикл от выбранной даты - дата создания

Логика подсчета показателя - Цикл до выбранной даты - дата встречи (факт)

Обратите внимание, что название полей с датами у разных клиентов могут отличаться. Выбор полей для данного фильтра нужно обсуждать с клиентом и далее проверять его корректную отработку.

На скрине видим, что цикл составляет 311 дней. Если открыть детализацию и посмотреть на столбцы с датами, то видно что это корректное значение - между датой создания лида и датой встречи (факт) прошло 311 дней.



С помощью данных фильтров можно строить показатели по циклу между датами, которые указаны в 1 сущности CRM. Это важно учитывать, тк если у клиента CRM в которой каждый этап воронки = отдельный CRM элемент, то построить цикл от встречи до сделки по такой логике не получится. Однако, если это CRM Амо или Битрикс24, то данная логика фильтрации будет хорошо работать, когда клиент в 1 карточке срм элемента указывает и обновляет все нужные даты.



Параметры фильтрации бывают 2 типов:

  • вводимые вручную (текст, регулярка)
  • выбираемые из выпадающего списка


Если для фильтра нужно изменить способ выбора параметра фильтрации (заменить выпадающий список на ручной ввод или наоборот) - обратитесь в тех.поддержку Smartis.


В Смартис можно создавать сквозные фильтры

Сквозной фильтр - фильтр, который создается через шаблон в консультации и позволяет фильтровать одновременно разные CRM сущности с полями, которые называются по-разному, но содержат одинаковые данные.

Например, у клиента есть сущности: Договор купли-продажи, Интерес, Визит, Договор бронирования. В этих сущностях есть признак "Отдел", но поля в сущностях называются по-разному: 

Название полей с Отделом:

Отдел - в Договор купли-продажи

Подразделение - в Интерес

Отдел после перевода - в Визит

Отдел - в Договор бронирования

=

Сквозной фильтр позволяет Объединить все эти фильтры в единый и назвать его общим названием - например, "Отдел".

Условия для фильтрации стандартные:

  1. равно

  2. не равно

  3. включает

  4. не включает

  5. не пустое значение

  6. пустое значение

Применяется один фильтр для всех 4 сущностей: Договор купли-продажи, Интерес, Визит, Договор бронирования

  1. "Отдел" и фильтрует все эти 4 сущности одновременно. 


  2. Из сквозного фильтра нельзя скрыть данные, которые "не нужны" (показываем всегда все значения)
