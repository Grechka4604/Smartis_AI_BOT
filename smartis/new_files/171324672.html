Основная информация

Это продукт, который позволяет верифицировать (проверять) обращения по заданным критериям, чтобы определить, является ли оно целевым, ориентируясь не на теги в колл-треккинге или CRM, а на наличие признаков продвижения обращения по воронке продаж. Верификация работает с обращениями из колл-трекинга и других сервисов обращений (то есть не с CRM).

Этот продукт помогает:

  • контролировать эффективность агентств-подрядчиков
  • находить первичные обращения (первичные по застройщику)
  • находить брошенных клиентов

Алгоритм верификации настраивается для каждого клиента индивидуально, поэтому алгоритм будет отличаться в зависимости от бизнес-процесса клиента. Пример алгоритма верификации: 

Верификация ежедневно обрабатывает весь поток обращений (кроме городских номеров и невалидных контактов) и выводить по ним итог, обозначающий качество обращений:

  • целевое
  • не целевое
  • условно-целевое
  • ? (спорный)
  • не определено (если обращение не проверифицировано)

Ежедневно по верификации доступен итог верификации за вчера и ранее. 
Верификация обрабатывает данные за столько дней в прошлое каждый день, сколько определено алгоритмом верифкации, созданным для клиента - это значит, что итог верификации по обращениям, которые поступили в этом периоде, будет пересчитываться.


Для клиента можно зафиксировать дату месяца, когда мы будем фиксировать итог по прошедшему месяцу и больше не обрабатываться обращения (например, если клиент сверяется с агентством и подводит итог каждый месяц 10 числа, можно указать эту дату. После 10 числа следующего месяца результаты верификации за прошедший месяц не будут изменяться).

Как настроить верификацию

  1. В разделе Администрирование → Интеграции подключить интеграцию "Верификация обращений"

  2. После нажатия "Подключить" откроется окно настроек интеграции, в котором нужно заполнить данные.

  3. Алгоритм - выбрать из выпадающего списка нужный (системный или пользовательский)

  4. Период возобновляемости клиента, дн. - Если вы перестали работать с конкретным клиентом - реклама может «Возобновить» его интерес по истечению настроенного количества дней отсутствия действий по этому клиенту. Укажите, чтобы не искать таких клиентов. Эта настройка позволяет выделить клиентов, по которым нужно возобновить рекламу. У разных застройщиков период может оличаться.


  7. Целевые теги - можно выбрать целевые теги из любого сервиса обращений, который подключен у клиента. Если нет явных признаков качества обращения - мы рассмотрим более детально обращения помеченные этими целевыми тегами. Любой из указанных тегов - повод более детально рассмотреть обращение. Список доступных фильтров отображается слева, справа отображается список сервисов обращений, в которых тег находится.


  8. Критерий брошенности клиента, дн. - определяет, через сколько дней отсутствия активности по клиенту его нужно считать брошенным. Если с клиентом нет взаимодействий на протяжении этого или более количества дней - он будет помечен брошенным. Поле заполняется произвольно, в зависимости от БП клиента

  9. Подтверждение интереса - можно выбрать CRM сущность, которая обозначает подтверждение интереса звонящего. То есть, если клиент продвинется до данного этапа воронки, то верификация засчитает подтверждение интереса. Выбирать можно из любых настроенных показателей в Smartis (то есть до настройки верификации показатель уже должен быть создан)

  10. Период создания интереса в минутах - Время, в течение которого в CRM, по итогу звонка, создается интерес (по умолчанию сутки от времени обращения). Здесь можно указать произвольное время.

  11. Продвижение до встречи - можно выбрать CRM сущность, которая обозначает продвижение клиента до встречи (показатель, который в Smartis обозначает сделку). То есть, если клиент продвинется до данного этапа воронки, то верификация засчитает, что он продвинулся до встречи. Выбирать можно из любых настроенных показателей в Smartis (то есть до настройки верификации показатель уже должен быть создан)

  12. Продвижение до брони - можно выбрать CRM сущность, которая обозначает продвижение клиента до брони (показатель, который в Smartis обозначает бронь). То есть, если клиент продвинется до данного этапа воронки, то верификация засчитает, что он продвинулся до брони. Выбирать можно из любых настроенных показателей в Smartis (то есть до настройки верификации показатель уже должен быть создан)

  13. Продвижение до сделки - можно выбрать CRM сущность, которая обозначает продвижение клиента до сделки (показатель, который в Smartis обозначает сделку). То есть, если клиент продвинется до данного этапа воронки, то верификация засчитает, что он продвинулся до сделки. Выбирать можно из любых настроенных показателей в Smartis (то есть до настройки верификации показатель уже должен быть создан)

Детализация данных верификации

В детализации данных по обращениям у клиентов с подключенной верификацией есть такой набор полей. 

  • Верификация: Итог - содержит информацию о качестве обращения (целевое, условно-целевое, сомнительное, не целевое или не определено). По клику на кнопку "спорный" можно оспорить результат верификации вручную - для этого нужно кликнуть "оспорить" и указать нужный вам итог верификации. По кнопке "проверено" - подтвердить оспоренный итог.
    При нажатии на эти кнопки система зафиксирует запрос на проверку итога верификации:


  • Верификация: Детали - содержит подробную информацию о том, почему присвоен такой итог верификации. Описывает как пришли к результату, опираясь на алгоритм клиента.

  • Верификация: Отражение в CRM - содержит ID CRM-элемента в Smartis, если он был создан в CRM по итогу данного обращения. По клику на ID открывается CJM этого клиента.

  • Верификация: Первичность активности - информация о первичности обращения (первичное, повторное, возобновленное и т.д.). Первичность определяется алгоритмом клиента, период возобновляемости отражен в настройках интеграции "Верификация"

  • Верификация: Даты совершенных визитов - если клиент с обращением совершал визиты в офис продаж, в столбце будут указаны даты совершенных визитов (опираясь на метрику Визит, указанную в настройках интеграции "Верификация".

  • Верификация: Объекты совершенных визитов - содержит информацию об объектах ЖК, по которым клиент совершил визит в офис продаж (если совершал).

  • Верификация: Интерес подтвержден? - содержит информацию о том, совершал ли клиент действие, которое подтверждает интерес к покупке (это действие указывается в настройках интеграции "Верификация" в поле Подтверждение интереса.

  • Верификация: Продвижение по воронке - содержит информацию о том, продвигается ли клиент по воронке продаж.

  • Верификация: Продвижение до встречи - содержит информацию, продвинулся ли клиент с этим обращением до встречи (продвижение до встречи = совершение клиентом действия, указанного при настройках интеграции в поле Продвижение до встречи)

  • Верификация: Продвижение до брони- содержит информацию, продвинулся ли клиент с этим обращением до встречи (продвижение до встречи = совершение клиентом действия, указанного при настройках интеграции в поле Продвижение до брони)

  • Верификация: Продвижение до сделки - содержит информацию, продвинулся ли клиент с этим обращением до встречи (продвижение до встречи = совершение клиентом действия, указанного при настройках интеграции в поле Продвижение до сделки)

  • Верификация: Дата верификации - последняя дата, когда это обращение верифицировалось.

  • Верификация: Дата оспаривания - дата, когда итог верификации оспаривали

  • Верификация: Дата подтверждения - дата, когда итог верификации подтверждали

  • Верификация: Подробные причины отказа - если клиент отказался от покупки и в CRM заполнены причины, они отобразятся в этом столбце. Поле, из которого нужно брать информацию о причинах отказа, указывается при настройках интеграции тут: Поле с причинами отказа в CRM (нормализованные значения)

Их можно добавить через меню добавления столбцов в детализации




Как проверить, что верификация работает корректно

  1. Контроль не проверифицированных обращений: если есть не проверифицированные обращения "за вчера", это означает, что не всё доверифицированно на данный момент. Это не норма, тк к началу рабочего дня верификация должна отрабатывать полностью за вчерашний день.
    Для контроля строим отчет:
    Показатель: Все обращения (или Все обращения (новый))
    Фильтр Верификация: Итог "не включает" [выбрать все итоги из выпадающего списка] + Фильтр     Person: ID персоны -"не пустое значение"
    Группировка любая, канал "Весь трафик", период - любой, включая вчерашний день. Проект "Все объекты".
    Если верификация работает корректно, то в отчете будет 0.
    У обращений, которые не проверифицированы, итог верификации "Не указано"

  2. Если мало проверифицированных обращений: Также можно мониторить полноту данных, которые приходят из сервиса обращений и из CRM. Для этого нужно построить отчет за период месяц, в группировке "День". желательно использовать визуализацию "график" для наглядности. Выбрать показатель "Все обращения" или обращения по конкретному сервису. Если вы видете резкие спады в динамике поступающих обращений - то возможно у вас проблема с полнотой данных из сервиса обращений, и это напрямую влияет на то, что верифицированных обращений стало меньше.

  3. Если мало обращений, которые верификация определила как целевые, условно-целевые и как "?" - проверить полноту CRM данных (как сырых, так и тех, которые в настройках интеграции "Верификация" у вас указаны как события, подтвержающие интерес и продвижение по воронке продаж.
    Проверить это можно также построив отчет по дням по всем необходимым метрикам и посмотреть, есть ли "провалы" в данных. Если есть, то проверить достоверность по сервису CRM. 

    Для клиентов с верификацией рекомендуется настроить технический дашборд мониторинга, в котором будут все эти отчеты.



  4. Что еще влияет на объем не проверифицированных обращений:
    - объем городских телефонов (обращения с городских не анализируются)
    - объем нормализованных телефонов (проверить корректность выполнения синхрона id 42, который должен приводить телефоны из системы клиента в формат Smartis:  79998887766)
    - адекватность телефонов (если телефон персоны - в черном списке, то верификация их не анализирует). Как это посмотреть - построить отчет по всем обращениям и добавить фильтр Person: id персоны - пустое значение. Также можно сразу добавлять фильтр Верификация: Итог "не включает" [выбрать все итоги из выпадающего списка] - так вы посмотрите информацию только про не проверифицированным обращениям.


    Добавить условие по мобильным номерам (как на скрине) - оставим только клиентов с пустым id персоны и при этом мобильном номере.


    Проверим, не в ЧС ли эти номера - для этого откроем детализацию выбранных обращений. Видим, что номера в ЧС - это значит, верификация работает корректно, а номера в ЧС - поэтому они и не верифицировались.
