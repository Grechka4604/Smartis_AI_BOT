Процесс окончательно не зафиксирован, поэтому могут быть внесены изменения в результате внедрения.

  • Все метрики и конкретные моменты будут указаны
  • Так же после внедрения будет представлена диаграмма процесса

Сценарии обработки задач

1. Сценарий — создание нового функционала

Инициализация задачи

  • Один из участников команды инициирует задачу на разработку нового функционала или доработку текущего.

Продуктовое исследование

  • Дополнительное обогащение задачи информацией

Проектирование решения

  • Системный аналитик пишет техническое задание (ТЗ) на разработку.

Разработка

  • Разработчик имеет право вернуть задачу на предыдущий этап в случае отсутствия необходимых компонентов ТЗ.
  • Разработчик реализует новый функционал согласно техническому заданию.
  • Важные моменты разработчик сохраняет в тикете.
  • Как только задача отгружается в stage или же передается в приемку (если задача принимается в dev) создается тикет на дополнение документации, который попадает в раздел “Новый функционал” в проект База знаний

Создание пользовательской документации

  • Системный аналитик имеет право запросить у разработчика дополнения технических аспектов задачи, в случае их отсутствия или неполноты.
  • Системный аналитик на основе технической документации (конечного содержимого тикета) разрабатывает пользовательскую документацию.
  • Тикеты на создание документации к новому функционалу являются наиболее приоритетными. (sla - 7 дней)

Валидация готовой документации

  • PM валидирует готовую документацию
  • PM имеет право запросить, пересмотр или дополнения документации в случае неудовлетворенности ее качеством

Ознакомление пользователей с новым функционалом

  • Продукт-менеджер знакомит пользователей с новым функционалом.

Пополнение системы обучения

  • Системный аналитик отправляет тикет на доску дополнения системы обучения.

2. Сценарий — дополнение текущей документации

Инициализация задачи

  • Любой участник команды может создать тикет на дополнение документации.

Валидация задачи

  • Задача попадает в общую очередь
  • Технический писатель валидирует запрос на корректность и необходимость.

Дополнение документации

  • Документация дополняется или обновляется в зависимости от запроса. Тут могут быть задачи различной направленности, выделяется три основных вида:
    • Post-View // тяжелое - задачи по сервису Post-View, требующие привлечения других людей для изучения. (sla - 14 р. дней)
    • Нужно дополнить - небольшие задачи, требующие небольшого дополнения в уже существующей документации. (sla - 7 р. дней)
    • Большие задачи - задачи требующие большое количество времени на иззучение и решение. (sla - 14 р. день)

Валидация готовой документации

  • PM валидирует готовую документацию
  • PM имеет право запросить, пересмотр или дополнения документации в случае неудовлетворенности ее качеством

Ознакомление с обновлениями

  • Если это пользовательская документация, пользователей информирует Product Manager.

Пополнение системы обучения

  • Если это пользовательская документация системный аналитик отправляет запрос на обновление обучающей базы.

Типы задач

  1. Новый функционал
    • Задачи из этого раздела относятся к тикетам, которые были созданы автоматически после выполнения задач, на дополнение функционала.
    • Данные задачи имеют наивысший приоритет по выполнению
    • SLA - 3 дня
  2. Нужно дополнить
    • Небольшие задачи, требующие незначительного дополнения в уже существующей документации
    • SLA - 7 дней
  3. Большие задачи
    • Задачи требующие большое количество времени на изучение и решение.
    • SLA - 14 дней
  4. Post-View
    • Задачи по сервису Post-View, требующие привлечения других людей для решения.
    • SLA - 14 дней

Основные метрики для оценки эффективности процесса

  1. Вся документация готова к ревью обновлений.
  2. Количество тикетов на дополнение < 5.
  3. Количество задач, в которых требующаяся информация уже лежит в Базе знаний < 1 в неделю.
  4. Количество задач из нового функционала, не требующие описания (для себя, чтобы понять все ли задачи на новый функционал требуют описания)
