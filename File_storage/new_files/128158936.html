Бэклог запросов состоит из 2 частей: сами доработки + список отчетов, который необходим колл-центру.

Ключевые моменты встречи:

1) Роли:
Проблема: Есть сотрудники которые должны помогать руководителям контролировать пропущенные,
мониторить статус коллег и иметь доступ к их отчетам, но этого нельзя сейчас сделать потому что отчеты доступны только рукоовдителям.

Сделали: добавили сотрудникам функцию доступа в онлайн-мониторинг, с возможностью менять статус коллегам, НО они не могут смотреть основные отчеты РМР и могут смотреть только онлайн-мониторинг по своей группе. Плюс, более того, чтобы увидеть данные по сотрудникам других групп в онлайн-мониторинге, этот сотрудник должен состоять в этих группах. При добавлении этого сотрудника в группы, которые им надо контролировать - необходимо отключить доступность номера телефона этого сотрудника. ЭТО СЛОЖНО, НЕУДОБНО, НЕПОНЯТНО. Простыми словами: никакой пользы для клиента доработка - не несет, т.к. отчеты сотрудник все равно не может формировать, а должен был бы.

2) Маршрутизация:
Проблема: звонки с Амурского приходят на Причальный, сотрудники не могут запомнить на какую группу каким образом им переводить звонок.
Сейчас: на Причальный звонок переводится через опцию разговора "вызов сценария" (стоит на кнопке 9),
потому что звонок на этом проекте должен равномерно распределяться между 3 группами брокеров и внутри группы брокеров (можно сделать только через сценарий).
По всем остальным проектам через обычный трансфер, через "#" вводят внутренний номер группы.

3) РМР:
а) Онлайн-мониторинг:
- По входящему потоку видим 2 цифры: принятые и необработанные. Отчеты не прозрачные, не можем провалиться в них. Для того чтобы посмотреть детально эти цифры - необходимо перейти в 2 других отчета и настроить необходимые фильтры с полным пониманием логики сборки отчетов. ЭТО НЕУДОБНО

б) Отчет по группам:
- Нет суммы всех звонков, приходится вручную считать общее количество звонков за определенный период, чтобы понять, что цифры в Онлайн-мониторинге и в Отчете по группам - сходятся.
- Формирование показателя "Пропущенные": Считается ли звонок пропущенным второй раз после не дозвона по всем группам брокеров (3 группы брокеров)? Хотят конкретно понимать сколько пропущенных звонков (а не вызовов) было по каждому брокеру.
- Столбцы "Принятые" и "Принятые трансферные": показатели в них считаются независимо друг от друга, или столбец "Принятые" является суммой всех принятых звонков группой?
- Брокеры замотивированы на звонки. Получают деньги за обработанные лиды. Руководитель колл-центра должен видеть: сколько звонков получил каждый брокер, равномерно ли распределяются звонки между брокерами, и в случае пропущенных звонков брокерами - видеть причину.
Кейс: Ком.дир брокера обращается к Руководителю отдела продаж Алине Ларионовой с жалобой, что у них нет звонков. Алина не знает где посмотреть так ли это и проверить причину.

в) Отчет по пропущенным:
- Клиент не понимает: откуда берутся цифры, плюс, данные в нем, не сходятся с данными по пропущенным в "Онлайн-мониторинге" в "Отчете по группам"?
- Как посмотреть Статистику по пропущенным звонкам в разрезе сотрудника/группы с причиной пропуска (звонок пропущен на уровне этого МП, звонок пропущен после ожидания на линии)?

г) Показатели необработанных звонков в "Онлайн-мониторинг" и "Отчет по пропущенным" - разные. Нет четкого понимания какие звонки остались необработанными и ждут прозвона со стороны сотрудников.

д) Эффективность сотрудников:
- Показатель "Принято" = "Звонки" = "Принятые в отчете по группам", а показатель "Пропущено" = "Вызовы" = "?" т.е. по факту сейчас непонятно, сколько всего пропущенных звонков и скольким людям из этих пропущенных - надо перезвонить?
