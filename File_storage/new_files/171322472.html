Основная информация

Это новая (с 2022 года) отраслевая СРМ для недвижимости. Они объединяют в себе управление продажами, коммуникациями, шахматкой, документооборот. Маркетинговая аналитика у них не развивается - мы помогаем закрыть эту задачу в партнерстве.

Мы общаемся с ними на разных уровнях, с нашей стороны курирует партнерство - Александр Калинин.

В Smartis с этой CRM нативная интеграция (работает по API и подключается без участия разработчиков). Из МакроCRM мы получаем всю информацию о клиентах - сделки, их статусы, контактные данные и другие параметры, которые для каждого покупателя заполняются в CRM.

Данные из CRM не содержат информацию о том, с какого источника (релкамного, органического) "пришел" клиент. На основе информации о рекламном источнике предыдущих касаний клиента с рекламой (просмотры медийки, сессии, обращения) строится аналитика по CRM данным в разрезе каналов, площадок, кампаний и т.д. То есть, в Smartis работает атрибуция CRM данных (в зависимости от выбранной Модели Атрибуции)


Из Macro CRM НЕ передаются удаленные пользователи.

Привязка интеграции к сайту

Для того, чтобы данные по MacroCRM  отобразились в статистике проекта в Smartis, необходимо привязать интеграцию к нужному проекту. Данные мы всегда привязываем к базовой единице - сайту.

  1. Для привязки необходимо пройти в раздел "Проекты и сайты", выбрать нужный сайт и зайти в его настройки. 
  2. В доступных для привязки интеграциях необходимо выбрать интеграцию  MacroCRM и привязать ее к сайту. 


  3. После того, как привязали интеграцию к сайту, никакие дополнительные действия не нужны - интеграция отобразится в разделе "Привязанные к сайту интеграции"

  4. Из интеграции c MacroCRM привязки по разным условиям для отдельных сайтов настроить невозможно - эти настройки выполняются в коннекторе Smartis.СRM, который также нужно привязать к сайту (выбрать его из раздела доступных интеграций и нажать "Привязать к сайту"). О том, как настроить привязки к сайт с помощью коннектора - читать тут: 6. Коннектор Smartis.CRM

Для корректной привязки к сайтам важно знать бизнес-логику клиента - как именно он хочет смотреть данные в Smartis и как он их делит на объекты. Например, некоторые клиенты выделяют как отдельный объект нежилые типы недвижимости - паркинг, кладовки, коммерческие помещения. В каких доп полях хранится нужная информация для привязки лучше уточнить заранее.
Также обратите внимание, что если из MacroCRM приходят данные по разным CRM сущностям, то привязку по полю сущности CRM нужно указать для всех нужных сущностей.

В макро у разных клиентов мы взаимодействуем с разными полями. Это значит, что нет унифицированных правил, по которым мы забираем какие-то данные из макро (тк поля у разных клиентов называются по-разному). К примеру, для определения объекта ЖК у одного клиента мы смотрим на поле "Интерес к проекту", а у другого это поле может называться "ЖК договора" или как-то иначе.

Показатели

При подключении интеграции создаются системные показатели (автоматиечски). Они будут отображаться в списке показателей в конструкторе отчетов, когда выбран проект, объект или сайт, к которому привязана интеграция. 
Показатели будут отображаться в отдельной категории - MacroCRM


MacroCRM: Сделки - содержит данные о договорах купли-продажи
MacroCRM: Встречи - содержит данные о встречах с клиентами
MacroCRM: Заявки - содержит данные о заявках в CRM
MacroCRM: Звонки  - содержит данные о звонках в CRM
MacroCRM: Задачи - содержит данные о задачах менеджеров продаж в CRM
MacroCRM: Планы продаж - содержит информацию о плане продаж


Фильтры для данных MacroCRM

Есть фильтры применимые только к данным из MacroCRM. Так как в основном это фильтры по доп.полям - их набор будет зависеть от того, какие доп.поля использует клиент у себя в MacroCRM. В поиске фильтров их можно найти по ключу "MacroCRM".

Обратите внимание, что при фильтрации по тегам в макро может быть такое, что когда выбрано несколько тегов, то в отчет будут попадать одни и те же сделки несколько раз. Само кол-во сделок уникальных по id останется прежним. Просто наличие каждого из перечисленных тегов будет засчитывать как бы 1 сделку.

Детализация данных MacroCRM

Для данных MacroCRM доступна детализация (исходные данные). Они такие же, как у других сервисов CRM.

Основные свойства, передаваемые в даных MacroCRM:

  • Данные о свойствах CRM-элементов: внешний ID, дата создания, тип CRM-элемента
  • Данные о клиенте: номер телефона, email, ФИО, статус на сейчас.
  • Данные о рекламном источнике обращения: канал, площадка, кампания (эти столбцы заполняются в зависимости от выбранной модели атрибуции)
  • Данные из доп полей: суммы сделок, объект (ЖК), ответственный менеджер, площадь, комнатность и другие параметры.

Достоверность MacroCRM

отображается в разделе Мониторинг системы. О его работе можно почитать тут: 9. Администрирование: Мониторинг системы

По данному сервису мы получаем информацию по таким сущностям:

  • встречи
  • задачи - если задач "у нас" больше, чем "всего" - это ок. Поэтому в таком случае мы считаем достоверность этой сущности 100%. Задачи не участвуют в аналитике и ключевых метриках, поэтому такие расхождения допустимы.
  • заявки
  • звонки
  • планы продаж - если планов продаж "у нас" больше, чем "всего" - это ок. Поэтому в таком случае мы считаем достоверность этой сущности 100%. Планы продаж не участвуют в аналитике и ключевых метриках, поэтому такие расхождения допустимы.
  • сделки - в контроле достоверности сделок - у сделок нет как таковой даты создания, поэтому мы ее проверяем по дате сделки deal_date (поэтому у вас может быть ситуация что сделки в отчете и в достоверности имеют разное количество)
     
  • Достоверность ретро данных по MacroCRM считается автоматически за последние 3 дня ежедневно.

    В информации о достоверности показано, сколько данных у нас и сколько данных в API MacroCRM, а также дата последнего обновления.
    Обновить данные по сервису из раздела Мониторинг системы нельзя (не добавляли такую возможность). По запросу можно доделать через обращение к разработчикам Smartis.

1.) можем ли мы автоматически брать из сервиса настройку часового пояса? как работает эта настройка?


Мы берем дату и время из данных полученных по API по всем сущностям и записываем как есть, как к нам приходит из API, без конвертации или обработки часового пояса, с той датой и временем как указано у клиента.
На данный момент автоматически не забираем настройку часового пояса из сервиса. Подобной настройки в пользовательских настройках интеграций так же нет.


Связанные методы:
В основном используется функция \strtotime
\App\Classes\Macro\Synchronizers\SynchronBase::getFormatedDateValue
Carbon::createFromTimestamp((int)$item['date_modified'])->toDateTimeString()


2.) можем ли мы вообще на своей стороне вносить правки, чтобы данные по сервису отображались в нужном нам часовом поясе?


Можем, но не будем. По всем сервисам. В интерфейсе мы отображаем все данные в едином часовом поясе.


3.) знаем ли мы, в каком часовом поясе приходят данные из сервиса?


Не знаем, берем даты сущностей как есть, как они приходят к нам по API.
