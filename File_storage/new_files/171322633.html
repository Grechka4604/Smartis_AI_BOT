Что такое?

В Smartis есть возможность просматривать детализированные (сырые, исходные) данные по некоторым сервисам:

  • CRM (как данные из нативынх интеграций, так и импортированые вручную или автоматическим импортом данные)
  • все сервисы обращений (колл-трекинг, сервисы квизов, сервисы заявок)
  • сессии (Посетители, которых мы собираем кодом, установленным на сайтах клиента)
  • микроконверсии (данные о поведении пользователя на сайте, которые мы собираем кодом, установленным на сайтах клиента)

По остальным сервисам сырые данные недоступны. При попытке построения отчета в формате "Исходные данные" по таким показателям отчет покажет ошибку:При построении отчета возникла ошибка. Для таблицы smartcallback_ch2.insights_ch1 отсутствует класс реализации получения исходных данных 

Для разных пользователей и отделов можно настроить разные уровни доступа к исходным данным (например, можно скрыть номера телефонов или суммы сделок для подрядчиков).

При просмотре исходных данных всегда отображается количество уникальных по ID элементов. Поэтому, если при построении отчета в группировке в итоге/значении по показателю вы видите например значение 100, а при открытии исходных данных по этому значению видите 90, это значит, что уникальных элементов по показателю 90. Отличие в таблице связано с тем, что в связи с привязками/другими настройками какие-то строки исходных данных дублируются (учитываются несколько раз). Для решения проблемы различий нужно проверить привязки и настройки показателя. 


Если в отчете выбрано несколько показателей с разными типами данных:


  1. Показатели с одним типом данных (например, CRM или сервисы обращений)  - в таком случае в Исходных данных будут все данные из выбранных показателей на одной странице
  2. Показатели с разными типами данных (показатели и CRM, и из сервисов обращений и другие) - под данные каждого типа создастся отдельная вкладка в отчете (на скрине - появилась вкладка для данных по обращениям)

  3. Показатели с разными типами данных + показатели, по которым исходных данных не существует: в таком случае отчет не построится и появится ошибка: При построении отчета возникла ошибка. Для таблицы smartcallback_ch2.insights_ch1 отсутствует класс реализации получения исходных данных

Чтобы отчет в исходных данных построился, необходимо убрать показатель, в котором используются данные, у которых нет Исходных данных. В примере на скрине - это показатель Я.Директ Клики

Способы просмотра исходных данных

  1. В конструкторе отчета через визуализацию "Исходные данные" (описано тут: ). Чтобы это сделать, выберите показатель и остальные настройки, нажмите иконку "Исходные данные" и постройте отчет.
  2. В конструкторе отчета через другие способы визуализации: Одноцифровой отчет, Таблица, Комплексный отчет

    Одноцифровой отчет -чтобы перейти в режим исходных данных, нужно кликнуть на плашку одноцифрового отчета. В сырых данных отобразятся все элементы, которые показывает одноцифровой отчет.

    Таблица - в режиме таблицы видно, какие данные можно посмотреть в детализации, а какие нет. У данных, которые можно посмотреть в исходном виде, есть пунктир под значениями в таблице:

    Каждое значение с пунктиром - кликабельно. То есть, в таблице можно смотреть исходные данные по каждой "ячейке" - строка + столбец, а также по строке ИТОГО. Если в таблице несколько показателей можно смотреть в исходном виде, то исходные данные будут показывать по клику данные по строке и по метрике, которая указана в заголовке столбца. Аналогично и с итогом - он рассчитывается для каждого показателя, то есть по столбцам.

    Комплексный отчет - здесь просмотр исходных данных также доступен только для таблицы (нижняя часть отчета). Логика просмотра исходных данных будет аналогична логике описанной в пункте выше про таблицу.
  3. В дашбордах: логика такая же, как и в конструкторе отчетов - по клику на значения в виджете открываются исходные данные 
    Одноцифровой отчет - чтобы перейти в режим исходных данных, нужно кликнуть на плашку одноцифрового отчета. 


    Таблица     - данные, подчеркнутые в таблице пунктиром, кликабельны - по клику откроется просмотр исходных данных.

Информация в исходных данных

Для того, чтобы изменить порядок столбцов в детализации, делаем так:

  1. В окне с выбором полей нажимаем Delete, чтобы удалить ВСЕ столбцы из детализации (то есть не должно остаться столбцов, отмеченных в списке галочкой)
  2. выбираем галочками нужные столбцы в нужном порядке
  3. каждый новый выбранный столбец будет добавляться в конец таблицы, таким образом можно выстроить их в нужном порядке
  4. набор столбцов сохранится для всех показателей с таким типом CRM элементов для пользователя в ЛК, под чьими доступами вы авторизовались. То есть каждый юзер может сам себе настроить так нужный порядок и набор полей в детализации.

Добавление дополнительных столбцов:

При добавлении новых полей в детализацию они автоматически закрепляются, если пользователь не снимает соответствующую галочку. Выбранные столбцы остаются доступными только для того, кто их добавил, обеспечивая персонализированный и удобный просмотр данных.

Сервисы обращений

Столбцы будем разбирать по порядку. Аналогичная визуализация как в Базе обращений, так и в любых других способах просмотра исходных данных по обращениям. Также разберем только основные столбцы, которые есть в исходных данных по обращениям.

  1. Smartis ID - содержит уникальный идентификатор, который присваивается обращению в Smartis. 
  2. Внешний ID - содержит уникальный идентификатор, который присваивается обращению в сервисе, из которого оно пришло. 
  3. Person ID - содержит уникальный идентификатор, который присваивается пермоне в Smartis. (о том, что такое персона, читать тут: )
  4. Разложение по МА - статус разложения по МА и присвоения Person ID обращению. Подробнее см. ниже. раздел Статус Разложение по МА. 
  5. Сервис - указывает, из какого сервиса пришло обращение
  6. Тип - тип обращения (например, входящий, исходящий, заявка, анкета и тд)
  7. Сайт - сайт, по которому пришло обращение (например звонок с сайта mgu.ru - значит, клиент был на сайте mgu.ru и позвонил/оставил обращение по телефону, которыйон увидел на этом сайте)
  8. Телефон клиента - номер телефона клиента в формате 7900000000. Если у клиента указано несколько контактов в CJM, то они будут перечислены через запятую.
  9. Запись разговора - актуально только для колл-трекингов (запись можно прослушать и скачать)
  10. Теги - теги, которые присвоили в сервисе обращений. Также теги можно редактировать вручную в Smartis - нужно кликнуть на окно выбора тегов и добавить/удалить теги
  11. Дата создания - когда обращение создано в Smartis
  12. Площадка - площадка, с которой поступило обращение
  13. Кампания в Smartis - кампания, в которую записано обращение в Smartis
  14. РК Треккинга - название кампании, которое мы получили из сервиса обращений
  15. Город - город, откуда пришло обращение
  16. Регион - регион, откуда пришло обращение
  17. utm_source, medium, campaign,content, term - в этих столбцах хранятся данные о utm метках обращения. Данный раздел будет пустой, если обращение пришло со статики (не подменный номер в колл-трекинге)
  18. Ссылающаяся страница 
  19. Страница входа - страница сайта, на которую заходил пользователь

MD5 телефона клиента/ email- хэшированный номер телефона или email

Кроме этих столбцов есть и другие - их набор зависит от параметров, которые в исходных данных передают разные сервисы обращений, а также от подключенных платных функционалов в личном кабинете.
Чтобы посмотреть, какие есть еще доступные данные, нужно кликнуть на "Добавить столбцы" и пролистать выпадающий список. Чтобы добавить столбцы в отображение, нужно кликнуть на название нужного столбца в выпадающем списке. Столбцы, которые выбраны для отображения, выделены жирным шрифтом и галочкой. После добавления столбца, он появится в конце (справа) в отображении. Если вы просматриваете обращения по разным сервисам одновременно, то при добавлении столбца какого-то 1 сервиса, данные в нем заполнятся только для обращений из этого сервиса.

на скрине показаны жирным - выбранные столбцы, обычным - столбцы, которые можно также отобразить в отчете.



Обратите внимание, что в детализации по обращениям не указывается дата конверсии и модель атрибуции, если это показатели не по моделям атрибуции. Это связано с тем, что на обычные обращения МА не влияет. Также работает и в ситуации, когда в детализации данных есть вкладка "обращения"  - например, если показатель который мы смотрим включает разные типы данных. Обратите внимание, что если ваш показатель включает и обычные обращения, и обращения по моделям атрибуции, модель атрибуции, дата конверсии и окно ретро отображаться будет.

Для обращений по моделям атрибуции эти данные будут отображаться при просмотре детализации при проваливании из любой группировки. Например, если вы построили отчет по каналам, и хотите просмотреть данные по 1 строке, то при проваливании вы увидите информацию о модели атрибуции.

Какая модель атрибуции будет отображаться при проваливании из дашборда (в том числе когда в отчете отображаются вкладки с типами данных - CRM, обращения и тд):

  • выбранная в дашборде
    • если для виджета не закреплена модель атрибуции
    • если для показателя не установлено перекрытие модели атрибуции
  • выбранная в виджете: 
    • если для виджета закреплена модель атрибуции
    • если для показателя при этом не установлено перекрытие модели атрибуции
    • в этом случае мы игнорируем модель атрибуции, выбранную в дашборде
  • выбранная в показателе:
    • если для показателя установлено перекрытие модели атрибуции
    • в этом случае мы игнорируем модель атрибуции, выбранную в дашборде и в виджете
  • несколько моделей атрибуции: в случае, когда у вас показатель построен с формулой, и для показателей в формуле закреплено перекрытие по разным моделям атрибуции.

CRM Данные

При просмотре CRM данных сверху отображается Модель атрибуции, выбранная для отчета или для показателя, если в показателе настроено перекрытие модели атрибуции.
Также отобразится дата конверсии - дата, выбранная для показателя. В рамках окна ретроспективы будут учтены касания до этой даты конверсии.

Эти данные будут отображаться при просмотре детализации при проваливании из любой группировки. Например, если вы построили отчет по каналам, и хотите просмотреть данные по 1 строке, то при проваливании вы увидите информацию о модели атрибуции.

Пример: в отчете выбрали МА последнее касание, 90 дней, не включая прямые касания. 

Провалимся в исходные данные по показателю Договор бронирования подписан: вверху указана МА как в отчете. 

Дата конверсии = дата подписания договора. Указана эта дата, тк в настройках показателя указали ее

Дата конверсии по умолчанию = дата создания. Однако если вы в показателе выбрали другую дату через фильтр "Календарь: дата отображения в отчете" и "Календарь: дата фильтрации периода", то отобразится дата конверсии = выбранная дата.


А теперь провалимся в показатель "Брони (тест)", для него мы настроили перекрытие МА внутри показателя. Здесь уже отображается МА из показателя.

Дату конверсии мы не меняли, поэтому осталась прежняя - дата подписания.

Вкладка с CRM данными будет всегда 1, даже если показатель включает разные типы CRM элементов. Это обосновано тем, что в детализации мы отображаем уникальные строки по id CRM элементов в смартис. То есть данные не дублируются.

У CRM данных набор столбцов в детализации зависит от полей, которые передаются или загружаются вручную из CRM клиента. Для разных CRM-элементов набор столбцов в детализации может отличаться. 
Есть минимальный набор, который необходим и есть у каждого клиента. Это поля, которые передаются из CRM, а также поля, которые заполняются на стороне Smartis:

  1. ID - содержит уникальный идентификатор, который присваивается CRM-элементу в Smartis. 
  2. ID в CRM - содержит уникальный идентификатор, который присваивается CRM-элементу в CRM. 
  3. Smartis Person ID - содержит уникальный идентификатор, который присваивается перcоне в Smartis. (о том, что такое персона, читать тут: )
  4. Разложение по МА - статус разложения по МА и присвоения Person ID обращению. Подробнее см. ниже. раздел Статус Разложение по МА.
  5. Тип - тип CRM-элемента (договор, интерес, активность и тд)
  6. Дата создания - когда элемент создан в Smartis
  7. Контакты клиента - номер(а) телефона клиента
  8. Канал, Площадка - столбцы, в которых указано к какому каналу и площадке атрибуцированы CRM элементы в Smartis. (если отчет построен в модели "линейное распределение", то каналов и площадок для 1 элемента может быть указано несколько, вес каждого канала будет указан в виде десятичной дроби). При этом значение веса в детализации у каждой площадки/канала/кампании итд должны совпадать с весом, который построится в обычном отчете по каналам/площадкам при тех же параметрах атрибуции.
  9. Проект - название ЖК, к которому относится CRM-элемент.
  10. Дату первого обращения - указывается дата первого обращения (звонка, заявки, заполненной анкеты, заполненной формы на сайте, лида ВК и тд) в рамках данного ЛК. 
  11. Дату первого касания - указывается дата первого касания с рекламой (кроме просмотров) в рамках данного ЛК. 

    Остальные поля CRM являются дополнительными, их набор зависит от:
    - особенностей CRM клиента
    - от того, какие данные клиент нам передает
    - от наличия некоторых платных функционалов Smartis (например, CDP)
    - от того, какой показатель выбран (например, у показателя Сделки могут быть столбцы Сумма сделки, Способ оплаты и тд. А у Активности или Интереса таких полей обычно нет)
    - добавляли ли мы доп поля клиенту через разработку. В интерфейсе смартис такие поля никак не помечаются и узнать, какие поля мы добавляем, можно только обратившись в консультацию или к документации по клиенту.

*Если у клиента в CRM добавляется новое поле, которое нужно отображать в детализации, необходимо обратиться в тех поддержку Smartis.

Также, последний набор выбранных столбцов для показателя сохраняется - если вы обновите страницу, переавторизуетесь в кабинете или вообще построите отчет заново - набор столбцов, отмеченных галочками, сохранится.

Также в исходных данных можно отображать кол-во касаний. На данный момент мы перешли на новый алгоритм анализа, и только в единичных клиентах используется местами старый анализ для обогащения. 
В скором времени старого анализа больше не будет

  • посчитать среднее значение кол-ва касаний можно через фильтр "Логика подсчета", но высчитываться будет по другому (новым анализом)

Статус Разложение по МА

Для обращений и CRM элементов в таблицах можно вывести столбец Разложение по МА. Таким  образом, можно увидеть, в каком состоянии находится поиск и присвоение Person ID обращению или элементу. 


 Возможные статусы и их описание:

'Не анализировалось' - Анализ и присвоение Person ID будет проведено в ближайшее время (как правило, не более чем в течении 2 дней).
'Успешно обработано' - Анализ и присвоение Person ID успешно проведено.
'Не нашли касания' - Не нашли касания - отнесли к "другое-остальное".
'Не нашли телефонных номеров клиента' - Не нашли телефонных номеров клиента. Если появятся номера, будет отправлено на повторный анализ.
'Переанализ (данные обновилось)' - Отправлено на повторный анализ в связи с обновлением данных.
'Переанализ с пересозданием персоны' - Отправлено на повторный анализ с пересозданием персоны.
'Слишком много запросов' - У данного элемента CRM неадекватное количество обращений. Вероятно имеющиеся контактные данные - не принадлежат конкретному человеку.
'Персона была найдена' - Персона найдена, осталось разложить по МА
'Повторное разложение по МА' - Отправлено на повторное разложение по МА. Персона остаётся без изменений.

Зачем это нужно и как этим пользоваться

Данная информация является в болшей степени справочной или служебной и служит для анализа состояния анализа и присвоения Person ID обращениям и CRM элементам. Также, на основе анализа значений этого поля в совокупности со значениями Person ID и идентификатора клиента можно выявить отклонения в процессе присвоения Person ID (т.е. понять, что что-то идет не так). Поэтому, в случае возникновения сомнений или явных ошибок в идентификаторах персон, весьма желательно прикладывать к тикету скрины с этой информацией (таблицу обращений или элементов, содержащую кроме основных данных также столбец Разложение по МА).   

Например, если Person ID = 0 и статус Не анализировалось, то в ближайшее время можно ожидать присвоения Person ID или изменения статуса, показывающее, почему сделать этого не удалось.

В то же время, если Person ID = 0 и статус Не анализировалось висит уже несколько дней, это говорит об ошибках или низкой производительности алгоритма и, возможно, требует вмешательства разработчиков.

Статусы 'Не нашли телефонных номеров клиента' или 'Слишком много запросов' объясняют отсутствие Person ID. Если вы убеждены, что для конкретного обращения или элемента это не соответствует действительности (статус ошибочный и Person ID должен быть определен) - ставьте тикет .

Таким образом, воможно не только определить в каком статусе находится присвоение Person ID, но и выявить отклонения в поведении системы.
