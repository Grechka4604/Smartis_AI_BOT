Объект в логике клиента - это ЖК либо Дилерский центр (для авто). Реже объект может быть не ЖК, а недвижимость по типам (жилая, коммерция, парковки, кладовки). В любом случае, объект предполагает выделение из всего объема данных клиента чего-то определенного, сгруппированого по какому-либо признаку.

Перед настройкой получаем от клиента:

    • список всех сайтов и лендингов, которые относятся к объекту
    • проверить, что на всех сайтах установлен код смартис (тут важно: именно на сайтах, тк на лендингах не всегда есть возможность установить наш код)
    • какие данные по сайту относятся к этому объекту (могут относиться все данные или только некоторые, например если у клиента 1 сайт по всем жк, то для выделения данных по 1 жк нужна информация о том, по какому признаку настроить фильтрацию данных с сайта для объекта)
    • какие сервисы колл-трекинга собирают данные по объекту? как называется логин колл-трекинга, в котором данные содержатся?
    • какие сервисы обращений (заявок, чатов, квизов) подключены к объекту? какие логины этих сервисов содержат эти данные?
    • какие рекламные кабинеты собирают данные по объекту? 
    • какие логины этих сервисов содержат эти данные?
  • если данные логины/кабинеты не подключены в смартис, то нужно их подключить
  • будут ли CRM данные по объекту? Если да, то по какому признаку из CRM нужно эти данные выделить из всего массива
  • Если данные по ЖК нужно выделять  зависимости от заполнения какого-то доп поля CRM, то нужна информация как в каждом CRM элементе называется это поле, а также какие варианты заполнения обозначают нужный нам объект. Если работаете с АМО CRM, то нужна информация, в каких воронках содержатся данные по нужному объекту.
  • какому отделу компании клиента должен быть предоставлен доступ к объекту? Какие уровни доступа нужны?
  • нужно ли корректировать фильтрацию ключевых показателей и других показателей в отчетах, с учетом нового объекта? (Пример: раньше были данные только по жилой недвижке, а теперь добавили нежилую недвижку и выделили ее в отдельный объект. Тогда для некоторых показателей скорее всего нужно будет разделение, например "сделки жилое" и "сделки нежилое")
  • В какие общие проекты нужно добавить объект? (часто аг-ва ведут конкретные объекты, либо есть деление по регионам - нужно всегда добавлять в такие проекты новый объект, иначе клиент будет терять данные)

Для того, чтобы выделить данные в отдельный объект:

1. На этом шаге получаем список всех сайтов и лендингов от клиента, которые должны относиться к объекту. Если проект будет состоять из сайтов, то нужны ВСЕ сайты и лендинги, которые будут относятся именно к этому ЖК (добавляем сайты в Смартис, формируем из них Объект). Если объект будет состоять из других объектов - проверить, что в этих объектах есть все нужные сайты. Проверяем, что на всех сайтах установлен код смартис.

2. Когда подключили все сайты и лендинги для объекта в смартис, нужно настроить фильтрацию данных по сайтам (для сайтов, по которым не все сессии нужно записывать в данный объект). Для этого в настройках интеграции SmartCallBack (она создастся автоматически после того, как вы добавите сайт в смартис) можно заполнить поля "Страница входа соответствует / Не соответствует регулярному выражению). Чаще всего это нужно когда у клиента есть общий сайт и на нем есть страницы отдельных ЖК и для учета по объекту нам нужны только сессии клиентов, которые зашли на сайт на страницу нужного ЖК.



На следующем шаге мы будем привязывать интеграции к сайтам. Тут важно, что объект может состоять из нескольких сайтов. Однако, это не значит что к каждому сайту в объекте мы должны привязывать интеграции:

  • CRM привязываются к 1 (любому) сайту в объекте
  • рекламные кабинеты, колл-трекинг и сервисы заявок чаще всего тоже привязываются к 1 сайту, но могут быть кейсы когда клиенту важно смотреть стату в разрезе сайтов по рекламе и обращениям и есть признак, по которому можно отфильтровать кампании по разным сайтам. Тогда рекламные кабинеты могут привязываться к разным сайтам в рамках объекта

4. Теперь нужно выделить данные по объекту из привязанных интеграций (если к объекту относятся только какие-то определенные данные из подключенных сервисов).
Как отделить звонки коллтрекинга по проекту: часто бывает так, что клиент ведет в 1 кабинете колл-трекинга кампании по нескольким ЖК. И в случае создания отдельного объекта нам понадобятся не все звонки из кабинета. Чтобы выделить только те звонки, которые относятся к нашему объекту ЖК, выбираем:
логин колл-трекинга, который содержит звонки по этому ЖК   + настроить условия привязки. 
Ниже приведен пример для CoMagic. Также аналогичная инструкция есть на страницах всех колл-трекингов и сервисов заявок, тк у некоторых сервисов возможности условий привязок отличаются.


Например, когда у клиента зонтичный бренд, а каждый ЖК - отдельная страница или раздел внутри одного сайта. В таком случае мы должны перейти в настройку привязкок в привязанной к сайту в Smartis интеграции. 

Пример: в логине "Застройщик" есть информация о звонках по ЖК1, ЖК2 и Бренду. Нам нужно сделать так, чтобы к сайту ЖК1 были привязаны только те обращения, которые поступают по этому ЖК.
Как это можно сделать:

  1. Если в названии рекламной кампании в CoMagic указаны названия ЖК, то можно прописать условие (в виде регулярного выражения) в поле Название РК соответствует регулярному выражению (учтите все возможные варианты написания - на русском и английском языке, сокращения и прочее)
  2. Если нужно привязать все рекламные каспании, кроме определенных - заполнить в виде регулярного выражения поле Название РК не соответствует регулярному выражению
  3. Выбрать конкретные рекламные кампании CoMagic в поле "Выбор конкретных кампаний" (с этим пунктом будьте осторожны, если у клиента добавятся новые рекламные кампании - условия привязки нужно будет актуализировать.  Поэтому вариант из пункта "а" более надежный
  4. Метка utm_campaign соответствует регулярному выражению - если нужная метка указывается в utm_campaign (и например не всегда указывается в названии кампании) то можно использовать этот способ. С помощью регулярного выражения нужно указать все возможные значения меток.
  5. Url страницы обращения соответствует регулярному выражению - подходит в случае, когда в названии кампании не указано название ЖК, но обращения по данному ЖК поступают на определенный сайт.  Url страницы обращения также указываем в виде регулярного выражения. 
  6. Привязка тегами - позволяет привязать обращения, у которых в CoMagic проставлены определенные теги. Из выпадающего списка можно выбрать 1 или несколько тегов

5. После того, как подключили сервисы обращений, подключаем рекламные кабинеты. Важно чтобы клиент вам предоставил ПОЛНЫЙ список всех кабинетов, в которых ведется реклама по данному объекту, иначе будут проблемы со сверками. После того, как вы подключите кабинеты, нужно будет пройти этап привязки интеграций к сайтам и скорее всего настройку фильтрации данных (для случаев, когда в 1 рекламном кабинете клиент ведет рекламу не только по нужному объекту). 

О том, как привязать к сайту интеграции и настроить фильтрацию, можно прочесть на странице по каждому сервису в confluence в соответствующем разделе: 2. Рекламные кабинеты
Обратите внимание, что для разных рекламных кабинетов есть отдельные страницы, где описаны возможности фильтрации данных для конкретного сервиса.

6. Привязываем коннектор CRM. Если ранее вы уже подключили CRM клиенту, то ничего нового создавать не нужно. Только подключить CRM и ее коннектор (если CRM доступна нативно для интеграции из интерфейса (Битрикс24, Амо, MicrosofDynamics, Macro CRM, или любой другой кастомной, где есть еще интеграция), то коннектор создается автоматически. Для кастомных CRM, или импортируемых данных, коннектор необходимо создавать вручную). Если у нас есть интеграция с основной CRM и сопровождающий коннектор, необходимо их обе привязать к сайту. 
Фильтрация данных CRM по объектам настраивается ТОЛЬКО в коннекторе CRM. В самой интеграции CRM ничего настраивать не нужно!

7. При необходимости - настраиваем фильтрацию ключевых показателей с учетом нового объекта (нужно не часто, но бывают кейсы когда например клиент стал получать в CRM данные по машиноместам и теперь выделил их в отдельный объект, в связи с чем появилась необходимость разделить какие-то показатели на показатели по Квартирам и Машиноместам)

8. Далее выполняем проверку корректности привязки данных к объекту. Для этого строим отчет по данному объекту:

  • сессии
  • все обращения
  • сырые данные CRM
    и смотрим, что в отчет НЕ попадают данные по другим объектам. Для сессий и обращений проверяйте, что данные в отчете соответствуют условиям фильтрации, которые вы настроили в привязках. также может быть такое, что что-то в привязках не учли - в таком случае корректируем привязки, если в отчет попадают лишние данные. Для привязки CRM данных если привязку делали по полям CRM элементов, то нужно проверить, что в элементах, которые попали в отчет, нужное нам поле содержит именно те данные, которые обозначают наш объект.
