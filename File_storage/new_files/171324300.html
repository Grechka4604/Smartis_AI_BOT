Основная информация

Comagic основная функция сервиса - это коллтрекинг, так же имеются омниканальные виджеты, чаты, рабочее место руководителя, сбор заявок из различных сервисов, телефония и сквозная аналитика (конечно не такая крутая, как у нас).

В Smartis можно подключить интеграцию с CoMagic в качестве сервиса обращений (то есть для получения из CoMagic заявок, звонков и чатов). Интеграция нативная, работает автоматически.

Данные из Comagic содержат информацию о том, с какого источника (рекламного, органического) "пришел" клиент (совершил обращение). На основе информации о рекламном источнике обращения строится аналитика по CRM данным в разрезе каналов, площадок, кампаний и т.д.
То есть, на основании данных об источниках обращений из CoMagic в Smartis работает атрибуция CRM данных (в зависимости от выбранной Модели Атрибуции)

Обновление данных по интеграции

Данные по интеграции после подключения автоматически забираем за 30 дней в прошлое, с момента подключения интеграции. Если нужно обновить другой период - воспользуйтесь ручным обновлением  тут: 9. Администрирование: Мониторинг системы

При обновлении важно иметь в виду, что у сервиса есть лимиты API - они распространяются не только на запросы от Smartis, но и на запросы других сервисов клиента, которые обращаются по API к CoMagic. Если лимитов клиента не хватает на корректную работу со Smartis (низкая достоверность, тк не можем забрать все данные из-за исчерпания лимитов) - нужно чтобы клиент обратился в тех.поддержку CoMagic и увеличил лимиты (это платно).
Справка про лимиты: 

Информация по лимитам отображается в разделе Администрирование → Мониторинг системы по клику на интеграцию Comagic


При ручной синхронизации данных можно выбрать, какие данные обновить:

  • звонки
  • заявки
  • чаты
  • чаты с сообщениями


    Чтобы сэкономить лимиты (особенно если обновляете большой период), выбирайте обновление Чатов. Так как при выборе "чаты с сообщениями" ваш запрос за большой период может израсходовать весь доступный суточный лимит запросов.

Если количество звонков в отчете отличается от кол-ва, указанного в мониторинге достоверности:

  • проверьте есть ли звонки Спам
  • проверьте наличие исходящих звонков
  • создайте тестовый сайт и привяжите к нему все данные интеграции без дополнительных условий - в таком случае данные должны сходиться с мониторингом.

Подключение интеграции

Интеграция с CoMagic нативная и доступна для подключения в несколько кликов. Данные из CoMagic забираются по API. 

У клиента может быть несколько кабинетов CM, необходимо делать интеграцию с каждым кабинетом. 1 интеграция = 1 кабинет. 

  1. Запросить доступы у клиента для подключения интеграции (также клиент может подключить сам, не передавая доступы вам). Нужен логин и пароль. 

    ВАЖНО! Лучше подключать интеграцию с администраторского аккаунта Comagic, так как только администратору доступны удаленные рекламные кампании. Если нет возможности подключить админский аккаунт, подключаем пользователя с полными правами. 


  2. В Smartis перейти в раздел Администрирование → Интеграции и в разделе "Доступные для подключения" выбрать интеграцию CoMagic 



  3. Нажать "Подключить" и заполнить данные для подключения. На этом этапе выбрать, нужно ли синхронизировать заявки и чаты (это обсуждается с клиентом).

    Какой сервис подключаем -

    ВАЖНО! В группе кампаний Новосистем есть 3 кабинета с разными брендами коллтрекинга Comagic, UIS, CallGear. С июля 2023 года бренд CM поглотил бренд UIS, но интеграция у нас сделана с комэджик. У них у всех отличаются IP. Поэтому очень важно выбрать сервис через указанное выше поле.

    Логин/Пароль аккаунта - это учетные данные пользователя, зарегистрированного в CoMagic, которому доступны данные всего аккаунта CoMagic или его отдельных сайтов/сегментов данных внутри кабинета колл-трекинга.

    Синхронизация заявок - При первом запуске синхронизации мы попробуем синхронизировать все сущности, но если API CoMagic сообщит что в этом аккаунте не используется компонент заявок - отключим их синхронизацию

    Синхронизация чатов - При первом запуске синхронизации мы попробуем синхронизировать все сущности, но если API CoMagic сообщит что в этом аккаунте не используется компонент чатов - отключим их синхронизацию

    Синхронизировать сообщения из чатов - Получение сообщений из чатов. Перед включением этой функции убедитесь, что лимитов API CoMagic достаточно для получения всех сообщений. Так же для включения данной функции необходимым условием является влюченная синхронизация чатов

    При включении синхронизации сообщений из чатов мы будем получать сообщения (историю переписки) по чатам из CoMagic. Если данная синхроницация будет выключена и включенной и включенной будет только синхронизация чатов, то мы не будем забирать сами сообщения из чатов, а будем синхронизировать только кол-во чатов.

    Для ранее настроенных интеграций нужно сразу указать корректные значения



  4. После того, как мы подключили интеграцию, она отобразится в списке "Подключенные интеграции". Если в интеграции есть ошибки, они отобразятся в иконке подключенной интеграции. 

Привязка интеграции к сайту

Для того, чтобы данные по обращениям отобразились в статистике проекта в Smartis, необходимо привязать интеграцию к нужному проекту. Данные мы всегда привязываем к базовой единице - сайту.

  1. Для привязки необходимо пройти в раздел "Проекты и сайты", выбрать нужный сайт и зайти в его настройки. 
  2. В доступных для привязки интеграциях необходимо выбрать интеграцию CoMagic и "привязать ее с логином" к сайту. Логин в данном случае - это сайт в CoMagic, внутри которого настраивается подмена и рекламные кампании.   

  3. Из выпадающего списка выбрать логин, по которому приходят обращения для выбранного сайта (можно выбрать только 1 логин, если к сайту нужно привязать несколько логинов - пункт 2 и 3 выполняются столько раз, сколько логинов нужно привязать, если нужно привязать все логины интеграции - выбрать в списке "Все данные этой интеграции"). После выбора логина интеграция привяжется к сайту. 
  4. Если нужно изменить логин, по которому получаете данные из CoMagic, можно в уже привязанной интеграции выбрать другой логин и подтвердить изменения.

  5. Иногда данные по разным проектам могут находится внутри одного сайта (логина) CoMagic. Например, когда у клиента зонтичный бренд, а каждый ЖК - отдельная страница или раздел внутри одного сайта. В таком случае мы должны перейти в настройку привязкок в привязанной к сайту в Smartis интеграции. 



    Пример: в логине "Застройщик" есть информация о звонках по ЖК1, ЖК2 и Бренду. Нам нужно сделать так, чтобы к сайту ЖК1 были привязаны только те обращения, которые поступают по этому ЖК.
    Как это можно сделать:
    1. Если в названии рекламной кампании в CoMagic указаны названия ЖК, то можно прописать условие (в виде регулярного выражения) в поле Название РК соответствует регулярному выражению (учтите все возможные варианты написания - на русском и английском языке, сокращения и прочее)
    2. Если нужно привязать все рекламные каспании, кроме определенных - заполнить в виде регулярного выражения поле Название РК не соответствует регулярному выражению
    3. Выбрать конкретные рекламные кампании CoMagic в поле "Выбор конкретных кампаний" (с этим пунктом будьте осторожны, если у клиента добавятся новые рекламные кампании - условия привязки нужно будет актуализировать.  Поэтому вариант из пункта "а" более надежный
    4. Метка utm_campaign соответствует регулярному выражению - если нужная метка указывается в utm_campaign (и например не всегда указывается в названии кампании) то можно использовать этот способ. С помощью регулярного выражения нужно указать все возможные значения меток.
    5. Url страницы обращения соответствует регулярному выражению - подходит в случае, когда в названии кампании не указано название ЖК, но обращения по данному ЖК поступают на определенный сайт.  Url страницы обращения также указываем в виде регулярного выражения. 
    • Привязка тегами - позволяет привязать обращения, у которых в CoMagic проставлены определенные теги. Из выпадающего списка можно выбрать 1 или несколько тегов
    • Привязка "Текст заявки соответствует регулярному выражению" - позволяет настроить привязки по полю Comagic:text с помощью регулярного выражения. 



      Применяется, если например у клиента мультибрендовый сайт alarm-brands.ru и разделить данные из комеджика по рк, utm, url нет возможности. При этом, в тексте заявки можно передавать информацию для привязки. 
       


      Негативные условия работают как сужающие. 

      положительные условия будут работать по логике ИЛИ (то есть добавлять в выборку данные), а отрицательные будут сужать выборку.
      Например если заполнено  названия РК соответствует регулярному выражению, названия РК НЕ соответствует регулярному выражению и теги, то к сайту привяжутся данные в которых есть выбранные РК, а также выбранные теги (при этом теги могут быть и в других РК, которые мы не указывали в выборке), при этом в них нету РК из поля названия РК НЕ соответствует регулярному выражению.

      формула: (Название РК соответствует регулярному выражению ИЛИ Теги) И Название РК не соответствует регулярному выражению

      Пример:
      Рк соответствует "спб"
      Рк НЕ соответствует "директ"
      Теги "в работе" - в отчет попадут все обращения из логина, которые:
      пришли по рк "спб" с любыми тегами
      пришли с тегами "в работе" по любы РК
      и не попадут все обращения, в названии которых есть "директ"


Показатели

При подключении интеграции создаются системные показатели (автоматиечски). Они будут отображаться в списке показателей в конструкторе отчетов, когда выбран проект, объект или сайт, к которому привязана интеграция. Также показатели CoMagic звонки, CoMagic заявки и CoMagic чаты добавляются в формулу для системного показателя "Все обращения (новый)"


  1. CoMagic звонки - отображает данные по звонкам (источник звонка определяется на основе данных рекламной кампании CoMagic, по которой пришел звонок)
  2. CoMagic заявки - отображает данные по заявкам (источник заявки определяется на основе данных рекламной кампании CoMagic, по которой пришла заявка)
  3. CoMagic чаты - отображает данные по чатам (источник заявки определяется на основе данных рекламной кампании CoMagic, по которой пришел чат)
  4. CoMagic звонки (по моделям атр.) - отображает данные по звонкам, которые строятся по моделям атрибуции- то есть источник звонка определяется по-разному в зависимости от выбранной в отчете модели атрибуции. Про модели читать тут: 4. Модели Атрибуции
  5. CoMagic заявки (по моделям атр.) - отображает данные по заявкам, которые строятся по моделям атрибуции- то есть источник заявки определяется по-разному в зависимости от выбранной в отчете модели атрибуции. Про модели читать тут: 4. Модели Атрибуции
  6. CoMagic чаты (по моделям атр.) - отображает данные по чатам, которые строятся по моделям атрибуции- то есть источник чата определяется по-разному в зависимости от выбранной в отчете модели атрибуции. Про модели читать тут: 4. Модели Атрибуции


Фильтры для данных CoMagic

Эти фильтры применимы только к данным из CoMagic

  • CoMagic: visitor_id - по id посетителя сайта, присваивается в CoMagic

  • CoMagic: «Качество обращения» - по качеству обращения, которое определяется в CoMagic (целевое, не целевое, вторичное, первичное и прочие)

  • CoMagic: Кампания (РК) - позволяет выбрать из списка конкретные РК из CoMagic

  • CoMagic: Название РК - фильтрует по названию РК в CoMagic

  • CoMagic: Статус звонка - по статусу звонка в CoMagic (потерянный или принятый)

  • CoMagic: Теги - фильтрует по тегам, указанным в CoMagic

  • CoMagic: тип звонка (в CoMagic) - по типам звонка, которые существуют у клиента в ЛК CoMagic

  • CoMagic: Тип звонка (в Smartis) - фильтрует входящие или исходящие звонки

  • CoMagic: Тип рекламной кампании - фильтрует по типу РК (статическая или динамическая)

При необходимости можно добавить и другие фильтры на основе доступных данных в CoMagic

Детализация данных CoMagic

Для данных CoMagic доступна детализация (исходные данные). Они такие же, как у других сервисов обращений - их можно просматривать в конструкторе отчетов или в данных по показателям обращений по клику из отчета или дашборда, а также в базе обращений.

Основные свойства, передаваемые в обращениях CoMagic:

  • Данные о свойствах обращения из CoMagic: внешний ID, сайт, дата создания, запись разговора, длительность разговора, теги, номер обращения в сервисе (первичность), качество обращения в сервисе и т.д.; 
  • Данные о клиенте: номер телефона, email, город, регион, устройство и т.д.;
  • Данные о рекламном источнике обращения: РК трекинга, параметры сессии, utm_метки, gclid, yclid, страница входа, страница обращения.

Привязка источника обращения CoMagic в Smartis

Когда мы получаем данные из СoMagic, они изначально не относятся ни к каким рекламным источникам - привязка происходит в Smartis (из СoMagic мы получаем названия рекламных кампаний и utm метки, на основе чего и выполняются привязки).

О том, как привязать обращения по рекламным кампаниям СoMagic к площадкам в Smartis читать тут: 2. Каналы и площадки

  • ВАЖНО! с 2023 года в CoMagic существуют две версии алгоритма колтрекинга (2.0 и 3.0), который возвращает в Апи значения РК в разных полях иногда с разными значениями. Мы сохраняем РК в формате "РК_3.0 / РК_2.0". Чтобы корректно привязывать РК к площадкам - нужно знать в какой версии работает CoMagic у клиента и опираться на ту часть до или после "/" в значении РК

При просмотре кампаний, которые нужно привязать, доступны фильтры:

  • Показывать непривязанные
  • показывать привязанные
  • Показывать выключенные

    Если ни один из фильтров не включен - отобразятся ВСЕ кампании. Если включать фильтры, то в отображении будут кампании из выбранных фильтров. Например, если вы включите показывать непривязанные - отобразятся  непривязанные кампании. А если добавите фильтр Показывать выключенные - то отобразятся непривязанные кампании, а также кампании которые выключены.
    Рядом с названием РК есть "кружок" - зеленый означает, что РК активна в комэджике. Серый означает, что РК выключена.

    Фильтр "показывать выключенные" работает по этому признаку (то есть отображает кампании с "серым знаком" из комэджик. 
    К самим выключенным кампаниям мы не добавляем приписку [выключена], данные из выключенных РК таким образом не создают новые кампании в смартис, а склеиваются с данными по первоначальной активной кампании, так как по сути это одна РК.

Если обращения из СoMagic не привязаны (автоматически по названию кампании или вручную), то они будут попадать в площадку "Другую рекламу" в отчете по площадкам/каналам.


В СJM клиента обращения из CoMagic отображаются так:

Отображаются:

  • тип обращения (звонки, заявки, чаты)
  • статус (принятый/пропущенный)
  • длительность разговора
  • телефон клиента и оператора
  • Канал и Площадка (на основании того, к какой площадке мы привязали данные обращения)
  • регион
  • город
  • РК треккинга - это название кампании в СoMagic. 
  • utm метки - мы их также получаем из СoMagic

  • Ссылающаяся страница - страница, с которой был совершен переход на сайт, с которого было совершено обращение

  • Страница входа - страница сайта, на которую зашел пользователь перед совершением обращения

  • Google client ID - id пользователя в Google 
  • Yandex client ID - id пользователя в Яндекс метрике 
  • При наличии тегов - отобразятся и теги. Теги можно проставить и в Smartis, но в таком случае в СoMagic теги не передадутся.

Сам по себе коллтрекинг CoMagic определяет местоположение клиента по IP-адресу. Smartis умеет обогащать эти данные за счет определения геолокации браузера на момент сессии. Благодаря такой синергии застройщики знают местоположение входящего лида через CoMagic в Smartis с точностью до 95%+ (98% по итогам июня 2023).


Достоверность CoMagic


отображается в разделе Мониторинг системы. О его работе можно почитать тут: 9. Администрирование: Мониторинг системы


Особенности достоверности по CoMagic:

  • У сервиса CoMagic также есть информация в блоке "Спец информация" - там указаны сведения о лимитах (общие лимиты по сервису, которые расходует не только Smartis).
  • Если низкая достоверность по Comagic, проверьте как расходуются лимиты. Может быть такое, что лимитов недостаточно и мы не можем забрать часть нужных данных. Увеличить лимиты API может только клиент с помощью менеджера CoMagic. Часто съедание лимитов связано с получением сообщений из чатов из CoMagic. В этом случае можно отключить синхронизацию чатов в настройках интеграции (мы будем получать кол-во чатов, но сообщеия из чатов приходить не будут)

Работа с часовыми поясами

smartis_stat.services
id = 4
title = 'CoMagic'
service_class = '\App\Classes\CoMagic\Core'

1.) можем ли мы автоматически брать из сервиса настройку часового пояса? как работает эта настройка?

Автоматическое получение и использование часового пояса клиента внесено прямо в коде приложения, подобной настройки в пользовательских настройках интеграции нет.
Мы получаем часовой пояс в формате Europe/Moscow, Europe/Saratov и т.д. из настроек аккаунта которые приходят к нам по API запросу и конвертируем дату и время в часовой пояс Europe/Moscow.

Связанные методы:
Используется в Звонках, Чатах и Заявках:
\App\Classes\CoMagic\Synchronizers\SynchronBase::applyAccountTimezone

Используется в достоверности:
\App\Classes\CoMagic\Api::getAccountInfoObject

2.) можем ли мы вообще на своей стороне вносить правки, чтобы данные по сервису отображались в нужном нам часовом поясе?

Можем, но не будем. По всем сервисам. В интерфейсе мы отображаем все данные в едином часовом поясе.

3.) знаем ли мы, в каком часовом поясе приходят данные из сервиса?

Знаем по данным настройки аккаунта, конвертируем часовой пояс клиента в часовой пояс Europe/Moscow.
