Принцип работы модели. 


Принцип работы модели единой команды нагляднее всего показать на примере продукта матчинг. 


  1. Есть задача подключить матчинг клиенту. У данного продукта есть метрика "процент матчинга", которая согласовывается с клиентом. 
    - метрика процент матчинга; 

Для успешной настройки продукта, требуется выполнение сопутствующих условий: 

       2. Подключение динамики в коллтрекинге. 

          - метрика процент охвата; 

       3. Автоматизация встроенных форм и охват сторонних коммуникаций.

           - метрика количество неотцифрованных форм (=0); 


Таким образом: 

  • есть конкретная метрика основного продукта, которая согласована с клиентом и по сути отражает выполняем мы договорённости или нет; 
  • и есть метрики каждого из продуктов в составе нашей экосистемы, которые в частности показывают "уровень здоровья" конкретного продукта и они в целом влияют на процент матчинга; 
    + выполнять задачу необходимо через понимание целей и бизнес-ценности для клиента. 

В данном примере - аналитика занимает центральное место в экосистеме, а коллтрекинг и SCB является поставщиком информации для аналитики. Что бьётся с нашим утверждением, что сервис аналитики не полезен без подключенного коллтрекинга и оцифровки всех каналов для обращения клиентов. 


Этот принцип далее переносится  на работу в единой команде. Т.е. возможны два варианта: 

  1. PM аналитики настраивает матчинг и поочереди ходит с запросами к направлениям, но это только его головная боль какой процент матчинга по клиенту → не наш вариант. 
  2. Команда ориентирована на достижение командой цели - матчинг не менее например, 85% и сотрудник каждого направления понимает как результат в его части влияет на достижение ключевой метрики по клиенту. 


Матчинг - наиболее простой для понимая принцип взаимосвязей. Исходя из запросов клиента могут быть менее очевидные зависимости. Однако работа в команде как раз позволит использовать экспертизу каждого из направления

для поиска таких пересечений и оценки влияния. 


Что конкретно делаем. 

  1. Формирование единой команды (в разрезе клиента)
  2. Анализ текущей ситуации (в разрезе клиента, на основании плана разбора клиентов)
    - выделяем пересекающиеся задачи на базовом уровне поддержки (или потенциально влияющие на другие направления); 
    - выделяем проекты развития клиента (подключение доп. продуктов, увеличение точности по текущим инструментам, подтверждение, что приносим пользу и др.); 
    - определяем потенциальную бизнес-ценность для клиента; 
  3. Подготовка QBR 
  4. Презентация QBR клиенту 
    - подтверждение проектов развития; 
  5. Выполнение проектов развития; 
  6. Фиксация результата проектов с клиентом (презентация по проекту, QBR и т.п.); 
  7. Выпуск кейса (внешний/внутренний)
  8. Анализ клиента и т.д. 



ВАЖНО: 

  • выше перечисленные пункты - это план, далее он будет обрастать процессами, интерфейсами взаимодействия, регламентами и т.п.; 
  • нам важно фиксировать как внешние так внутренние кейсы, чтобы накапливать базу вариантов использования экосистемы; 
  • возможны варианты работы единой команды когда клиент использует все наши продукты и команда состоит только из сотрудников поддержки и варианта, когда мы например клиенту 
    допродаём аналитику или SCB, или коллтрекинг и тогда в единую команду подключается сотрудник отдела продаж;
