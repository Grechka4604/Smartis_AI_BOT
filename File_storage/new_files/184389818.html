Блок 1. Проверяем построение отчета, который будем валидировать. Следует: 

  1. Воспроизвести отчет глазами клиента (с тем же периодом, моделью атрибуции, группировкой)
  2. Проверить настройки виджета (закрепленные МА, период и т.п.), если смотрим уже готовый отчет


Блок 2. Проверка поступления поствью данных 

  1. Строим отчет в конструкторе по показателю "Просмотры медийных размещений".
    Добавляем фильтр: Медийка: Сервис Postview и выбираем нужный сервис (если нужно проверить циан - выбираем циан, если выбираем адривер - добавляем фильтр по площадке).

  2. Смотрим динамику просмотров за период по дням (оцениваем, есть ли провалы в данных, есть ли данные в целом).
    !Если данных нет, но они должны быть - создаем тикет на баги, запрашиваем лог данных (при необходимости)

Важно знать! Дополнительная информация по просмотрам:
- если за месяц общее количество просмотров до 10-20 млн просмотров - могут быть точечные изменения, но точно не глобальные
- в просмотрах есть ритм нового месяца (1-5 дней начала месяца может быть раскачка и данных в этот период может быть меньше, чем в другие дни)
- дополнительно можно проверить бюджет (мало ли срезали бюджет и это повлияло на просмотры), при этой проверке ВАЖНО, чтобы бюджет был внесен корректно

Блок 3. Возвращаемся к Поствью отчету 

  1. Возвращаемся к поствью отчету по площадкам, смотрим воронку и оцениваем эффект площадки на показателях.

Важно знать! Что такое эффект от поствью? Ожидаем увидеть рост конверсий х 2 - х 15 (на показателях С поствью он должен быть)

Если сомневаемся или не видим эффекта, то переходим к Блоку 4. 


Блок 4. Валидация слоя обращений

  1. Проверяем площадки и зашитые utm-метки в них
  2. Смотрим обращения и их рк треккинга например, в обращениях по циан медийка должны быть обращения с метками форматов ,но никак не cpa, а в площадке циан базы - не должно быть обращений с метками медийки (паралакс и другие типы медийки)

  3. Смотрим привязки РК треккинга к площадкам, чтобы проверить корректность связок, при необходимости корректруем
  4. Строим отчет по посетителям, группировка "Площадки". Цель: найти топы по площадкам

  5. Так как на медийку влияет охват коллтрекинга, мы выбираем звонки comagic/calltouch и только площадки топы, по которым будут обращения, смотрим сколько было звонков и сколько из них имеют не пустое значение gclid. Сначала просто смотрим звонки в количественном значении, затем звонки без google client id (обязательно брать площадки, которые ведут трафик
    Можно создать показатель: охват динамическим коллтрекингом. Плохие показатели (маленький % охвата) говорит о том, что хорошей связки не будет. Добавить скины построения показателя Охвата






    Дополнительно можно посмотреть динамику, как работали с охватом (скрин ниже). на этом этапе клиенту можно подсветить рекомендации по связке данных



Блок 5. Дополнительный блок по Cian

Валидировать подключение "Cian Mobile, так как Cian Mobile лучше матчит данные. Выбираем показатель "просмотры медийких размещений", добавляем фильтр: Медийка: Сервис Postview в сервисах должен быть Cian Mobile, если он подключен. Если его выбрать нельзя - сервис не подключен. Для подключения шаги описаны в Confluence
