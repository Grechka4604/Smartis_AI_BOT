Цель - обеспечить поставку ценности клиенту от внедренной сквозной аналитики, а так же обеспечить использование продукта сквозной аналитики клиентом на постоянно основе. Передать менеджеру поддержки всю необходимую информацию по клиенту для его дальнейшего онбординга. 

Участники процесса: Project-менеджер внедрения, Project-менеджер поддержки.

Необязательные: Руководитель отдела внедрения, Руководитель отдела поддержки. 

Инициатор: Project-менеджер внедрения

Инфо поле: Asana, conf, телеграмм


Результат/что является продуктом процесса: Project - менеджер поддержки взял на себя клиента, знает все нюансы работы с ним и понимает, как дальше двигаться по проекту.

Владелец процесса: Руководитель отдела внедрения/поддержки.

Статус процесса: Этап поддержки. 


После того, как клиент в письменном виде подтвердил, что внедрение завершено, начал пользоваться продуктом и готов перейти на поддержку с полной абонентской оплатой, происходит передача проекта из отдела внедрения на поддержку. 


 

 Регламент передачи из внедрения на поддержку.

  1.  Состоялась внутренняя встреча между ПМ Внедрения и ПМ Поддержки. 

          На который были проговорены и зафиксированы в Asana (в отдельной задаче). Шаблон https://app.asana.com/0/1206864152384341/1207379081825745 :

шаблон заполнения карточки в асана:  https://app.asana.com/0/1206864152384341/1206875756181594/f

  • Сотрудник отдела внедрения передал информацию по бизнес-задачам и целям клиента.
  • Сотрудник передал информацию по активным ЖК (неактивным ЖК, если есть)
  • Сотрудник рассказал про активные интеграции, подключенные в кабинете.
  • Сотрудник подробно рассказал про CRM, коллтрекинг их особенности. При необходимости отдал схемы бизнес-процесса клиента, модель данных. Дополнительно остановился на распределении данных между проектами. 
  • Сотрудник рассказал про особенности логики подсчета целевых метрик + приложил в задачу Бриф/документацию, согласованную с клиентом.
  • Сотрудник рассказал про контактные лица (внутри компании клиента  и все агентства). Выделил основного ЛПР.
  • Сотрудник зафиксировал и рассказал другие особенности клиента (Кастомные доработки по кабинету, дополнительные отчеты или ТЗ)
  • Сотрудник подробно рассказал про подписанные документы, рассказал из чего складывается ежемесячная абонентская плата по клиенту.
  • Сотрудник обозначил способ доставки закрывающих документов, кому отправляются счета и кто ответственный за оплату.
  • Сотрудник передал все задачи по клиенту и перечислил их в задаче по передаче.

       2. В основном проекте в asana на вкладке "Обзор" зафиксирована информация по шаблону https://app.asana.com/0/1206864152384341/1206875756181594.

  • Контактные лица (ФИО)
  • NDA
  • Описание компании
  • Документооборот\специфика:
  • РА (если есть список РА)
  • Коробка(лицензия)/облако
  • Информация по целям
  • Вложения по проекту

       3. В основном проекте в асане зафиксированы и описаны все задачи, которые менеджер должен подхватить:

  • Задачи на разработку (с полным описанием) 
  • Текущие задачи по клиенту, который менеджер должен подхватить
  • Задачи, которые касаются дополнительных доработок (списком, даже если уже закрыты)
  • Все постмиты с клиентом

      4. Менеджер поддержки должен ходить на еженедельную встречу, которую ранее согласовал менеджер внедрения. Если данные встречи актуальны для клиента. 

      5. Менеджер поддержки добавлен в телеграмм чат/ в переписку в почту.

      6. Необходимо брать менеджера поддержки на презентацию решения для клиента и познакомить с клиентом или создать другую трехстороннюю встречу знакомство.

      7. На встрече знакомство нужно проговорить с клиентом цели на поддержку.

ВНИМАНИЕ! Встреча по передаче проекта обязательно должна быть записана. 

Нюансы, на которые должен обратить внимание менеджер поддержки, при принятии проекта: 

  1. Зафиксирован процесс по занесению бюджетов? Если да, то каким способом и кто это делает? Если нет, то почему? 
  2. Если ли понимание, какие отчеты клиент смотрит ? Какие метрики для клиента в "приоритете"? 
  3. Если CRM кастомная, была ли передана и понятна информация по сущностям, способу интеграции, к кому идти, если "не работает", как сверять, как часто сверять? 
  4. Если есть доп. продукты, было ли описание как и зачем клиент их "покупал" ?
