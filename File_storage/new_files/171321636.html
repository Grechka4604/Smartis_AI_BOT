Общая информация

В Smartis есть раздел для мониторинга системы - он позволяет посмотреть, насколько полно мы забираем данные из интегрированных систем (для контроля достоверности) и обновлять данные по сервисам.

Раздел можно открыть по ссылке в облаке: https://my.smartis.bi/admin/monitoring или из раздела Администрирование→ Мониторинг системы

В этом разделе в разбивке по дням и по сервисам/интеграциям можно посмотреть, насколько полные данные по каждой из подключенных интеграций.

Мониторинг системы позволяет более детально изучить информацию о достоверности.
Достоверность - соответствие данных в БД Smartis и данных в API интеграции. Хорошая достоверность 97% и выше. Статистику по достоверности видно справа от меню Smartis.

В строке "Достоверность" отображается средняя достоверность по всем сервисам клиента. Этот процент кликабельный - по клику открывается средняя достоверность по сервисам клиента на данный момент.

Рядом с комментариями по сервисам при наличии ошибок есть знак "..." - при наведении курсора на него откроется список статусов или ошибок.

Если достоверность по сервису низакая, нужно зайти в раздел Мониторинг системы и выяснить причины. В облаке по ссылке https://my.smartis.bi/admin/monitoring или из раздела Администрирование→ Мониторинг системы


В меню достоверности в navbar не учитываются результаты достоверности по архивным и удаленным интеграциям, также как и в мониторинге достоверности


Достоверность по удаленным аккаунтам окрашивается в серый цвет.


Достоверность по сервисам

В правом верхнем углу можно выбрать интересующий период (не более 180 дней одновременно) - и мониторинг отобразит данные за выбранные дни. Если выбрать больше 180 ьдней - будет ошибка.

в столбце "Сервисы" перечислены все сервисы, интеграции с которыми есть у клиента в ЛК. напротив названия сервиса указана средняя достоверность по сервису за каждый день. То есть, если у клиента 5 интеграций с ЯД, то в строке напротив сервиса Яндекс Директ будет средняя достоверность по его 5 интеграциям. 
Дни в отчете кликабельны - если кликнуть на какой-либо день, то отобразится информация о сервисе справа на странице:

  • кол-во интеграций с сервисом
  • статусы по подсчету достоверности (только у Яндекс Директ)



Название сервиса кликабельно - по клику на него открывается меню справа: 

Оно содержит информацию об активных интеграциях с сервисом, синхронах по этому сервису и окно обновления данных (про него ниже)

У сервиса CoMagic также есть информация в блоке "Спец информация" - там указаны сведения о лимитах (общие лимиты по сервису, которые расходует не только Smartis).

Достоверность по отдельным интеграциям в каждом сервисе

Чтобы посмотреть достоверность по отдельным интеграциям, нужно кликнуть на иконку стрелки слева от названия сервиса - откроется список интеграций.

Дни по каждой интеграции тоже кликабельны: по клику на день открывается справка справа на странице:
Здесь указан: 

  • общий % достоверности по этой интеграции
  • подробная информация о достоверности, в разбивке по сущностям, которые получаем из интеграции. Например, для АМОCRM - по сделкам и контактам. Для разный сервисов набор будет отличаться
  • когда обновляли достоверность по этой сущности за выбранный день
  • сколько данных у нас
  • сколько данных в API (во внешней системе)


У каждой сущности достоверность может отличаться, поэтому чтобы рассчитать достоверность по интеграции за 1 день, мы считаем среднее арифметическое (без нулей) процента достоверности по всем сущностям. Здесь важно отметить, что 100% в качестве среднего арифметического мы указываем только если по всем сущностям процент достоверности = 100.


Если название интеграции в списке интеграций красным шрифтом - это значит, что по интеграции есть ошибка и данные по ней не поступают (доступы неверные, или ошибка токена).

По таким интеграциям достоверность будет 0.
Также в списке не отображаются

  • удаленные интеграции
  • интеграции, которые находятся в архиве
  • интеграции, по которым еще ни разу не получали данные

В ситуации когда у клиента нет ни одной активной интеграции по сервису - выводится описание: "Не было ни одной активной интеграции по этому сервису за выбранный период либо они удалены". Мониторинг при этом будет пустым.



Название интеграций также кликабельно: по клику отображается меню справа

В меню показана интеграция (как она выглядит в разделе интеграций), окно для обновления данных и информация о синхронах


Если интеграция с ошибкой (горит красным) - то отобразится, что у интеграции есть ошибка


В информации по синхронам доступны такие данные:

  • id синхрона
  • название синхрона
  • периодичность его работы
  • статус (активен/не активен)



Ручное обновление данных

Для сервиса и отдельных интеграций доступна функция обновления данных.

Она доступна:

  • по клику на название сервиса
  • по клику на название интеграции
  • по клику на день в строке достоверности сервиса
  • по клику на день в строке достоверности интеграции


При обновлении "данных + достоверности" по всем CRM интеграциям, автоматически выстраивается такой порядок обновления сущностей:

  • контакты

  • сущности/справочники (список доп. полей, пользователей, etc...) мы возьмём один раз

Если обновляются данные за несколько дней, то порядок обновления будет такой:

  • 1.01 контакты
  • 2.01 контакты
  • 1.01 сущности/справочники
  • 2.01 сущности/справочники



Можно обновить:

  • отдельно достоверность (без обновления данных) - подходит для случаев, когда данные есть полностью, а достоверность отображается некоректно
  • достоверность + данные - подходит для случаев, когда нужно перезабрать данные из интеграции и обновить по ним достоверность


По клику на название сервиса или интеграции можно выбрать период, за который нужно обновить данные, а также сущности, по которым нужно обновить.
Период выбирать обязательно, сущности можно не выбирать - в таком случае данные обновятся по всем сущностям. Этот функционал позволяет забрать данные по недавно подключенным интеграциям (новым) за тот период, который необходим. 
ВАЖНО! рекомендуем не выбирать большой период обновления сразу (например, 1-3 года), а обновлять по месяцам - если будут ошибки, так будет проще отладить проблему.

Если вы обновляете по клику на Интеграцию - обновятся данные только в рамках интеграции. Если по клику на название сервиса - то данные обновятся по всему сервису, то есть по всем интеграциям сервиса за выбранный период.

Если вы запускаете ручное обновление данных без выбора конкретных сущностей по CRM интеграциям, то по умолчанию сначала обновятся контакты, а потом остальные сущности. Благодаря этому мы избегаем ситуации, когда контакт появится в системе позже сделки/визита/брони и это помешает анализу.



Чтобы обновить данные точечно по проблемным дням - лучше кликнуть на день в списке сервиса или в списке интеграции.
В таком случае данные можно обновить или за выбранный день по выбранным сущностям, или также указать период


После того, как вы нажали "Обновить" -  данные добавятся в очередь на обновление. Посмотреть статус обновления можно по иконке (смотреть скрин)

Если обновляется только достоверность, то статус в очереди будет: "Данные: завершено, достоверность: в процессе". Если обновляете данные + достоверность, то статус после запуска будет "Данные: в процессе, достоверность: в процессе"

Пока обновляется интеграция, в мониторинге по выбранным дням будет отображаться иконка круговых стрелок.

Статус обновления также можно посмотреть по клику на иконку запусков ручных операций:


Когда обновление завершено, статус будет как на скрине "Данные: завершено, Достоверность: завершено"

Это значит, что данные обновлены(если обновляли данные+достоверность) и достоверность за нужный период пересчитана.

В достоверности по CRM мы НЕ учитываем удаленные сущности. Это значит, что если сущность помечена как удаленная в смартис, и удалена в CRM клиента, то при подсчете достоверности мы не будем считать такие сущности. То есть мы всегда сравниваем Неудаленные в CRM элементы и не помеченные как удаленные CRM элементы у нас.
Это применимо к интеграциям:

  • Microsoft Dynamics
  • Amo CRM
  • Macro
  • 1С
  • CRM Endpoint

В Битрикс24 могут удаляться только Контакты, но они не имеют поля "Удалён" или чего-то похожего. Так как они не влияют на отчеты, то в случаях когда "у нас" контактов больше чем "всего", мы указываем достоверность 100%.

Лиды и сделки в Битрикс 24 в последний раз превышали значение "всего" в 2023-03-13, затем только в 22 году. Поэтому для них указанную выше логику учета удаления не реализовывали.

Также обновлять данные может автоматический контроль достоверности 


Что делать, если по сервису низкая достоверность

  1. Убедиться, что все нужные интеграции по сервису активны (у них нет ошибок) - для этого можно кликнуть на название сервиса в списке и увидеть список и иконки его интеграций. Также можно раскрыть список интеграций сервиса и проверить, что никакие названия интеграций не подсвечены красным. Если в интеграции ошибка - исправить ее в разделе Интеграции (переподключить, обновить и тд)
  2. Если проблема в каких-то конкретных днях - раскрыть список интеграций с сервисом и посмотреть, какие статусы автоматического контроля обновлений указаны (если "!" - автоматика больше не будет пытаться обновить данные, если "песочные часы" - автоматика в процессе работы и нужно подождать.
  3. Если в проблемном дне "!" - можно попробовать обновить данные за этот день вручную и дождаться завершения обновления достоверности. Если это не помогает (достоверность не изменилась) - обратиться в тех.поддержку Smartis.
  4. Посмотреть, по каким сущностям проблема - например, может быть такое что среди 5 сущностей, которые поступают из интеграции, не хватает данных только по какой-то одной. В таком случае можно обновить данные только по этой сущности
  5. Если достоверность низкая из-за того, что "у нас" данных больше чем "всего", проверить в отчете - если в отчете данных столько же, сколько в строке "всего" - значит, нужно обновить просто достоверность, без обновления данных, за этот день.
  6. Если у вас много дней по Яндекс Директ в большом периоде, где достоверность низкая. автоматическое обновление эти данные уже не будет обновлять, можно использовать Smart обновление данных: Система сама обновит данные и пересчитает достоверность везде, где она меньше 97%.Вам надо только указать интересующий вас период. Smart обновление доступна по клику на название сервиса или иъотдельной интеграции.
  7. Если у вас низкая достоверность по Маркетинговым касаниям, то в первую очередь нужно проверить как настроены МК. Между показателями могут быть пересечения в настройках - из-за чего будет конфликт, и МК будут создаваться не полностью. Изучите построение показателей по МК и проверьте их фильтры. Если пересечений нет - обратиться в тех.поддержку Smartis. Если есть - исправить и обновить маркетинговые касания.
  8. Если это сервис - Comagic, проверьте как расходуются лимиты. Может быть такое, что лимитов недостаточно и мы не можем забрать часть нужных данных. Увеличить лимиты API может только клиент с помощью менеджера CoMagic. Часто съедание лимитов связано с получением сообщений из чатов из CoMagic.
  9. Если при обновлении достоверности в мониторинге вы видите ошибку, пришлите текст ошибки в чат консультации для уточнения проблемы. Если выяснится, что проблема связана с багом - необходимо будет поставить тикет в баги.


Причины, почему достоверность не отображается (пустое поле)

  1. За просматриваемый период не получали данные по сервису/интеграции (то есть интеграцию подключили позже, в другом периоде)
  2. По интеграции еще не забирали данные (для новых интеграций). В таком случае нужно просто вручную обновить данные за нужный период по интеграции.
  3. Достоверность для данного сервиса не включена (есть шаблонная задача в консультацию по отображению достоверности по новому сервису https://app.asana.com/0/1204486395777830/1205284749626323/f)

Произошел технический сбой (в таком случае достоверности может не быть по нескольким сервисам у нескольких клиентов) - в таком случае нужно обратиться в тех. поддержку.


Как работает мониторинг системы (достоверность) коробки в облаке (когда для клиента коробки создан технический кабинет в облаке)

  • по интеграциям, которые подключены только в коробке, достоверность считается в коробке, но передается в облако

  • Отличить интеграции, которые созданы в коробке, от других интеграций можно по специфичному названию integration #...

  • в облаке интеграции, по которым отображается достоверность из коробки, НЕ СУЩЕСТВУЮТ (а значит, обновить достоверность по ним в облаке не получится)
  • в коробочных клиентах есть интеграция для мониторинга полноты переноса сессий из облака в коробку - "Для On-premise: SmartCallBack - оцифровка форм сайта". Она отображает, сколько сессий в облаке и сколько сессий в коробке. 100% достоверности означает, что все сессии из облака переданы в коробку. Если %достоверности ниже, значит часть сессий не передалась. Чтобы перенести недостающие сессии из облака в коробку, можно воспользоваться ручным обновлением данных как и по другим сервисам (обновить просто достоверность или достоверность + данные).
  • для Postview сервисов в коробке и облаке достоверность разная: в коробке отображается полнота переноса данных из облака в коробку, а в облаке отображается полнота полученных данных из интеграции с сервисом Postview.

Особенности достоверности по разным сервисам

Достоверность Яндекс.Директ.

Описание: Система каждый день рассчитывает достоверность данных за предыдущий день по следующим сущностям: расходы и ключевые слова. Мы обеспечиваем сходимость данных на 99%.

Пример: если сегодня 15.02.2023, то достоверность будет посчитана за 14.02.2023, и т.д.

При работе ручного обновления обрабатываются ручные операции, созданные за последнюю неделю

Исключения: для клиента ЮСИ, был реализован отдельный период подсчета достоверности данных - каждые 2 часа. И за сегодняшний день могут быть данные по достоверности меньше 99%. Яндекс на своей стороне анализирует данные и они могут меняться, и так как наши алгоритмы растянуты во времени, то с момента получения данных и до момента подсчета достоверности - данные могут измениться (Вся информация об актуальности данных в Яндексе)

По достоверности Яндекс Директ есть также сведения об истории запроса данных. Здесь отображается, по каким аккаунтам ЯД внутри логина запрашивали информацию, сколько попыток обновления было, а также информация об ошибках. 

Это полезно, когда у вас достоверность низкая, но не понятно, из-за чего. Например, могут быть ситуации когда аккаунт ЯД удален, но так как у логина есть к нему доступ, данные продолжаем запрашивать и получаем ошибку. Пример такой ситуации на скрине.


Достоверность переноса сессий из облака в коробку 

Описание: В коробках и в облачном кабинете коробочного клиента можно мониторить насколько полным был перенос сессий из облачного кабинета клиента. Достоверность отображается в сервисе SmartCallBack

Пример: Достоверность доступна сегодня за вчерашний день. если сегодня 15.02.2023, то достоверность будет посчитана за 14.02.2023, и т.д.

Обновить данные по этому сервису вручную через Мониторинг системы нельзя. В случае низкой достоверности нужно обратиться в консультацию/тех поддержку Smartis.

Дашборд мониторинга достоверности в grafana http://grafana.smartis.bi/d/DEYk2B87k/dostovernost?orgId=1&from=now-30d&to=now

Анализ данных: В коробочных клиентах также доступна интеграция "Анализ данных", которая показывает процент данных из CRM и сервисов обращений, проанализированных по персонам. Данные, не содержащие валидные контакты или содержащие аномальные идентификаторы, не учитываются в анализе. Достоверность анализа должна быть не менее 97%

если по клиенту не считается анализ, то нужно включить флажок is_ma_enabled
